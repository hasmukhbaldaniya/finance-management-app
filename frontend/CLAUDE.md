@AGENTS.md

## Overview

Next.js (App Router, TypeScript, `src/` directory, import alias `@/*`), styled with Tailwind CSS 4 + shadcn/ui. Implements login/forgot-password (`user-stories/001-authentication.md`), the 5-step company registration wizard (`user-stories/002-organization-signup.md`), the post-login header navigation (`user-stories/003-header-navigation.md` — its switch-organization screen is no longer implemented, see `SessionContext` below for why), Grade Management (`user-stories/004-grade-management.md`), Department Management (`user-stories/005-department-management.md`), Roles & Privileges Management (`user-stories/006-roles-and-privileges-management.md`), the owner-only Associated Organizations network (`user-stories/007-associated-organizations-network.md`), and the 4-step Invite Employee wizard, Story 1 of 3 in the Employee Management epic (`user-stories/008-employee-invitation.md`). Structure is modeled on `zp-frontend`'s conventions (see `../../nextjs/zp-frontend/CLAUDE.md`), scaled down to what this app actually has — folders aren't pre-created until there's real content for them.

## Project Structure

```
src/
├── apis/                # API layer — one subfolder per domain (currently: auth/)
├── app/                 # Next.js App Router routes, split into (public)/(private) groups
├── components/          # Feature/shared components + shadcn primitives (ui/)
├── contexts/            # React context providers, one <Name>Context/ folder each
├── hooks/               # Shared hooks (currently: useInfiniteScroll.ts)
├── stories/             # Storybook stories for every component, one <PascalCaseName>/ folder each
├── types/               # TypeScript types per domain
└── utils/               # constants/, helpers/, apiManager/
```
`src/hooks/` holds one hook so far: `useInfiniteScroll.ts` — created once Grade and Department Management both needed the identical "load the next page when the user scrolls to the bottom" behavior (an `IntersectionObserver` watching a sentinel `<div>`, no "Load more" button); Roles & Privileges reuses it too, unchanged. It reads `onLoadMore`/`hasMore`/`isLoading` via a ref updated in an effect (not during render — `react-hooks/refs` flags writing a ref synchronously in the render body) so the observer itself is created once per mount instead of being torn down and recreated on every fetch. Reuse this for every future paginated list (Employees next), don't re-implement it inline. No `src/icons/` yet — not created until there's a real shared icon component to put in it (see `zp-frontend/CLAUDE.md`'s own layer description for what it's for when that day comes).

### `src/apis/` — API layer

One subfolder per domain, one file per endpoint: `src/apis/auth/` (login/logout/me/forgot-password/registration endpoints — registration's endpoints are still `auth/*` since they're all under the backend's `/api/auth/registrations/*` namespace), `src/apis/organization/` (GST-availability check only — no more `listMine.api.ts`/`src/apis/user/`, both removed along with `GET /api/organizations/mine`/`PATCH /api/users/me/active-organization` per the User → Employee merge, see `backend/CLAUDE.md`), `src/apis/grade/`, `src/apis/department/`, `src/apis/role/` (each: list/create/update/updateStatus/delete/listMembers, one file each, matching the backend's `/api/grades*`/`/api/departments*`/`/api/roles*` routes one-to-one — `department/` and `role/` are straight copies of `grade/` with the noun swapped; `role/create.api.ts`/`update.api.ts` additionally take a `privileges: PrivilegeKey[]` argument), `src/apis/associated-organization/` (`list.api.ts`/`updateStatus.api.ts` only — no create/update/delete, matching the backend's read-mostly `/api/associated-organizations*` surface), `src/apis/project/` (`list.api.ts`/`create.api.ts`, for Step 2's Projects picker + inline "+ New Project"), `src/apis/airline/` (`list.api.ts` only, for Step 3's Airline picker), and `src/apis/employee/`, the largest domain — split across two stories: `008`'s wizard mutations (`create.api.ts`, `updateCompanyAccess.api.ts`, `addFfNumbers.api.ts`, `saveApprovals.api.ts`, `sendInvite.api.ts`) plus `listForPicker.api.ts` for the approver picker (`GET /api/employees/approvers` — moved off the bare `/employees` path once `009` needed that for its own listing, see `backend/CLAUDE.md`'s Employee Listing section); and `009`'s own `list.api.ts` (the main `GET /api/employees`), `resend.api.ts`, `updateStatus.api.ts`, `getDetail.api.ts`, and `updateBasicInfo.api.ts` (the last two power the Edit page). Every `.api.ts` file just calls `apiCall`/`postJson` from `src/utils/apiManager/apiManager.ts` and returns its typed response. Every domain folder has an `index.ts` barrel; import from the domain barrel (`@/apis/auth`, `@/apis/organization`, `@/apis/grade`, `@/apis/department`, `@/apis/role`, `@/apis/associated-organization`, `@/apis/employee`, `@/apis/project`, `@/apis/airline`), not individual `.api.ts` files, outside the domain itself. No query-key layer yet — this project doesn't use a data-fetching/cache library (no React Query), so there's nothing to key. If one gets added later, add `src/apis/queryKeys/<domain>.queryKeys.ts` to match `zp-frontend`'s convention at that point.

**Server-driven search/sort/pagination screens (Grade/Department Management, and their siblings to come) reset loading/error state synchronously at the top of an effect before refetching** — the standard shape for every future org-scoped CRUD screen (`src/app/(private)/company-settings/grades/page.tsx` and `.../departments/page.tsx` are the reference implementations — near-identical, `departments/page.tsx` is a straight copy with the noun swapped). `eslint.config.mjs` turns off `react-hooks/set-state-in-effect` project-wide for exactly this reason: without a data-fetching library, "fetch on a dependency change, resetting loading/error first" is the correct pattern (React's own docs list data-fetching as a valid effect use), and the rule's concern is an extra render pass, not a bug. Don't silence it per-line instead of relying on this — the whole point of turning it off centrally is that every future CRUD screen hits the exact same shape.

### `src/app/` — Routes

Two route groups (groups don't affect the URL — `/login` and `/dashboard` are unchanged):
- `src/app/(public)/` — unauthenticated screens (`login/`, `forgot-password/` and its steps, `register/` and its steps). No route protection.
- `src/app/(private)/` — authenticated screens: `dashboard/`, `trips/`, `claims/`, `approvals/`, `finance/`, `reports/`, `help/`, `profile/`, and `company-settings/{employees,categories,roles-privileges,grades,departments,associated-organizations}/` (no more `organizations/` — its switch-active-organization screen was removed, see `SessionContext` above). Only `categories/` is still a bare placeholder page (renders `<ComingSoon title="..." />` from `src/components/coming-soon.tsx`) — see `user-stories/003-header-navigation.md`'s Out of Scope for the original list of placeholders, now shrinking as each CRUD story ships. `employees/page.tsx` is now `009`'s real Employee Listing (search/status-filter/sort/pagination table, replacing the old `ComingSoon` + bridge "Invite Employee" button). `company-settings/grades/page.tsx`, `.../departments/page.tsx`, and `.../roles-privileges/page.tsx` are the reference implementations for every other org-scoped CRUD screen — see `src/components/grade/`, `src/components/department/`, and `src/components/role/` (`form-dialog.tsx`/`members-dialog.tsx`/`status-dialog.tsx` each) for the Add/Edit-with-Delete-inside, Members-list, and toggle-confirmation dialog patterns to reuse. `role/` additionally has `view-dialog.tsx` (read-only, for the two seeded default roles — Edit/Delete/toggle are all disabled for them, both in the UI and re-enforced server-side) and `privilege-checkbox-list.tsx` (the shared 8-item checkbox group both `form-dialog.tsx` and `view-dialog.tsx` render, backed by `src/utils/constants/privilege.constant.ts` — kept byte-identical to the backend's copy). **The status toggle always opens a confirmation dialog before changing anything** — `grade/status-dialog.tsx`/`department/status-dialog.tsx`/`role/status-dialog.tsx` — there is no optimistic/instant toggle anywhere in this codebase, even though [004](../user-stories/004-grade-management.md)'s story doc originally specced one; that decision was superseded after implementation (see that doc's Open Questions), and Department/Roles were both built with the confirmation dialog from the start.

  **`company-settings/employees/page.tsx`'s Suspend/Activate toggle is the one deliberate exception to "always confirm"** — per [009](../user-stories/009-employee-listing.md)'s own Flow, only Suspend opens a confirmation dialog (`src/components/employee/status-dialog.tsx`); clicking Activate on a Suspended row applies immediately with no dialog, since re-enabling access is the lower-risk direction. This is intentionally asymmetric, not a bug or an inconsistency with Grade/Department/Role's always-confirm pattern — don't "fix" it to match those without being asked. The listing also **excludes the logged-in caller's own row** entirely (added at explicit request, not in `009`'s original doc) — an admin manages their own account elsewhere, not through this table.

  **`company-settings/employees/[id]/edit/page.tsx`** is the Listing's Edit action — [009](../user-stories/009-employee-listing.md) explicitly defers this screen's field-by-field spec ("reuses [008](../user-stories/008-employee-invitation.md)'s rules verbatim... left for implementation time"), so it's built as a near-duplicate of `employees/invite/page.tsx`: same `SectionCard`/`NativeSelect`/`StepNav` structure, same validation, but pre-filled from `GET /api/employees/:id` and every write call updates instead of creates — no terminal "Send Invite" step, since Resend Invitation is its own separate row action on the listing, not part of editing. Given how much these two pages share, if a third near-identical form ever appears, that's the signal to extract the shared section markup into components rather than copy-pasting a third time — two is still an acceptable duplication, per this codebase's general "don't abstract until it hurts" bias.

  **`company-settings/employees/invite/page.tsx`** is a single-page form, not a multi-route wizard and not a list+dialog CRUD screen. [008](../user-stories/008-employee-invitation.md) originally specced this as a 4-step wizard (separate routes per step, gated by Continue/Back, each step's fields locked until the previous step's API call succeeded) — that was built first, then explicitly superseded at the user's request: all 4 sections (Basic Information, Company Access, FF Numbers, Access & Approval) render simultaneously on one page via `SectionCard` (`src/components/employee-invite/`), every field in every section is editable at any time in any order, and nothing is saved until a single "Send Invite" button at the bottom. See [008](../user-stories/008-employee-invitation.md)'s Open Questions for the full writeup — the per-step API endpoints themselves (`create`/`company-access`/`ff-numbers`/`approvals`/`send-invite`) are unchanged, only the frontend navigation model changed. No `EmployeeInviteContext`/`layout.tsx`/`StepShell` exist anymore (deleted when this collapsed to one page) — all wizard state is plain `useState` local to the single page component, since there's no cross-route state to persist. A left-rail `StepNav` (`src/components/employee-invite/step-nav.tsx`, `md:` and up only) lists the 4 sections as numbered steps and scroll-jumps to each `SectionCard` (matched by `id`); it's purely a navigation aid with an `IntersectionObserver` highlighting the in-view section — it does NOT gate editing or submission (the single-page "everything editable at once" model is the whole point of the redesign), so don't add step-completion locking to it.

  On submit, the page runs the 5 API calls in sequence (create → company-access → ff-numbers, only if any row was filled → approvals → send-invite), tracking the created `employeeId` in local state so a failed retry after the employee already exists doesn't re-call `createEmployee` (which would 409 on the now-duplicate email). The `createEmployee` call has its own nested try/catch specifically to map its 409s to the right inline field (email/contact number/Employee ID) — don't widen that mapping to cover the later calls' errors, none of which are field-specific in this UI.

  **Exception**: `company-settings/associated-organizations/page.tsx`'s Status column is a colored pill (green "Active"/muted "Disabled"), not a `Switch`, but it's still clickable and still instant/optimistic (flip immediately, revert-with-toast on failure), *not* confirmation-gated — this is not an oversight, it's what [007](../user-stories/007-associated-organizations-network.md) actually specifies (written and unchanged since before the Grade/Department/Role confirmation-dialog pattern existed) and nobody's asked to bring it in line since. If asked to "make it consistent with the other toggles," that's the one-line fix (swap the pill for a `Switch` + `status-dialog.tsx` like the others) — but don't do it unprompted, it's a deliberate per-story difference, not a bug. This page is also the **first screen in the app gated by role** — it doesn't render its own auth check; it relies on the header simply not showing its nav link for non-owners (see `header.tsx` below) and the backend 403ing directly if a non-owner reaches the URL/API anyway.

  Its filter UI is a deliberate one-off, redesigned after initial implementation against approved reference screenshots (see [007](../user-stories/007-associated-organizations-network.md)'s Open Questions) — don't copy this pattern onto Grade/Department/Role without a similar explicit design reference: a funnel icon (top-right of the title, `FunnelIcon`/`XIcon` from `@phosphor-icons/react`) toggles a per-column filter row that appears directly inside the table's `<TableHeader>` as a second `<TableRow>` — four debounced text `Input`s (Organization Name, Employee Name, Email Address, Contact No.) plus two plain native `<select>` dropdowns (Registration, Status; single-value, "All" + the field's fixed options — no shadcn Select primitive exists yet, so these are hand-styled natives, not worth adding a new primitive for two two-option/three-option dropdowns). Closing the filter row via the X **clears every filter**, it's not just a visibility toggle. Only two of the seven columns (Organization Name, Employee Name) are sortable — the rest render as plain, non-interactive headers, also per the reference design. The `MultiSelectFilter` component originally built for this page (multi-select checkboxes via `DropdownMenuCheckboxItem`) was deleted once the design moved to single-select dropdowns — if a future story needs a real multi-select filter, `DropdownMenuCheckboxItem`'s `closeOnClick: false` default is still the right primitive to reach for, just rebuild it rather than expecting it to still exist.

  Gated by `src/proxy.ts`'s `matcher`; any new authenticated route must be added under `(private)/` **and** to that matcher array. `(private)/layout.tsx` wraps every page in `SessionProvider` (see `src/contexts/`), which does the one `getMe()` fetch for the whole private area and renders the shared `Header` — individual pages must not re-fetch `getMe()` or render their own header/logout, they consume `useSession()` instead (see `dashboard/page.tsx` for the pattern). `useSession()` also exposes `isOwner` (from `GET /me`'s `isOwner` field) — the only place in the frontend that knows whether the current user owns their active organization; `Header` uses it to conditionally show the Associated Organizations (Network) sub-header link, and any future owner-gated UI should read it from here rather than re-deriving it.

Route paths are centralized in `src/utils/constants/route.constant.ts` (`ROUTES` object) — never hardcode a path in `href`, `router.push`/`replace`, or a `new URL(...)` call.

Route protection: `src/proxy.ts` (Next 16 renamed Middleware to Proxy; it defaults to the Node.js runtime, not Edge) verifies the session JWT itself via `jsonwebtoken` before allowing access to matched routes — it doesn't just check cookie presence. `AUTH_COOKIE_NAME` and `JWT_SECRET` env vars must match the backend's exactly (see `.env.example`); a mismatch fails safe (redirect to `/login`) but locks out every user if only one side is updated. `JWT_SECRET` has no fallback — the app throws at startup if it's unset, same as the backend — so an insecure default can't accidentally ship to production.

### `src/components/` — Components

- `src/components/ui/` — shadcn/ui primitives, copied (not npm-installed) via `npx shadcn add <component>` (or the `shadcn` MCP server in the root `.mcp.json`). Kebab-case filenames — this is shadcn's own convention, don't rename to fight the tool's update path. `dropdown-menu.tsx` is base-ui-backed (`@base-ui/react/menu`) like `button.tsx`/`input.tsx` — its subcomponents (`DropdownMenuItem`, `DropdownMenuTrigger`, etc.) render as a custom element via the `render` prop (base-ui's equivalent of Radix's `asChild`), e.g. `<DropdownMenuItem render={<Link href={...}>Label</Link>} />`, not `asChild` + a child element. Its icons were swapped from the shadcn CLI's default `lucide-react` to `@phosphor-icons/react` after adding it, matching this project's icon convention below — always check a freshly-added shadcn file for stray `lucide-react` imports. **`DropdownMenuLabel` must be a descendant of `DropdownMenuGroup`** — base-ui's underlying `Menu.GroupLabel` throws `"Menu group parts must be used within <Menu.Group>"` if rendered standalone; wrap it (and the items it labels) in a `DropdownMenuGroup`, or use a plain styled `<div>` instead if the text isn't actually labeling a group of items below it (see `header.tsx`'s Profile menu for the latter). `table.tsx`/`dialog.tsx`/`switch.tsx` (added for Grade Management) and `checkbox.tsx` (added for Roles & Privileges, its own privilege checklist) are the same base-ui-backed style — `dialog.tsx` and `checkbox.tsx` both needed the same `lucide-react` → `@phosphor-icons/react` icon swap as `dropdown-menu.tsx` did; check for it every time `npx shadcn add` pulls in a new primitive, it isn't a one-off.
- Truly cross-feature components (`auth-card.tsx`, `password-input.tsx`, `logo.tsx`, `header.tsx`, `coming-soon.tsx`) stay flat in `src/components/` — kebab-case filenames, not `zp-frontend`'s `PascalCase/<Name>.component.tsx`. `header.tsx` is the one shared component rendered outside `(public)`/`(private)` page bodies — it's mounted once by `(private)/layout.tsx`'s `SessionProvider`, not by individual pages. Two distinct nav patterns live in it, chosen per what the item represents: **Company Settings** is a plain `Link` (not a `DropdownMenu`) whose destinations render as a second nav row — a sub-header — directly under the main header whenever `pathname.startsWith("/company-settings")`, since they're peer pages a user switches between, not a transient action list. **Profile** stays a `DropdownMenu` popup, since its contents (identity display, View Profile, Logout) are a one-off action set, not pages to navigate between. Don't reach for `DropdownMenu` by default for every "item with children" — pick the sub-header pattern when the children are sibling routes.
- **Components unique to one feature get their own `<feature>/` folder** — `src/components/grade/`, `src/components/department/` (each: `form-dialog,members-dialog,status-dialog`), `src/components/role/` (those three plus `view-dialog.tsx` and `privilege-checkbox-list.tsx`), `src/components/employee-invite/` (`section-card.tsx` — a title/description + bordered-card wrapper, one per section on the single-page Invite Employee form (see `src/app/` above); `native-select.tsx` — a hand-styled native `<select>` matching `Input`'s height/border/radius, justified here despite the general preference against new UI abstractions because the form needs it for ~8 dropdowns and no shadcn Select primitive exists yet; `step-nav.tsx` — the left-rail scroll-jump nav), and `src/components/employee/` (`status-dialog.tsx` — the Suspend confirmation dialog only; unlike `grade/department/role`'s `status-dialog.tsx`, this one is never opened for the Activate direction, see `src/app/` above) — imported as `GradeFormDialog` from `@/components/grade/form-dialog`, `SectionCard`/`NativeSelect` from `@/components/employee-invite/section-card`/`native-select`, `EmployeeStatusDialog` from `@/components/employee/status-dialog`, etc. — the feature name lives in the folder, not repeated in the filename. This revisits the flat-components decision above: it made sense with one feature area (auth) and a handful of files, but stopped making sense once Grade/Department/Roles each grew several tightly-coupled dialog components. Follow this shape whenever a feature needs more than one component of its own, not something to defer indefinitely.
- Every component's story — `ui/` primitives included — lives under `src/stories/components/<PascalCaseName>/<PascalCaseName>.stories.tsx` (e.g. `src/stories/components/Button/Button.stories.tsx`, `src/stories/components/AuthCard/AuthCard.stories.tsx`), one directory per component, importing the component via its `@/components/...` path rather than a relative import — never colocated next to the component file itself, even for shadcn-managed `ui/` files. Required for every genuinely shared/reusable component (not one-off page-specific markup), covering its meaningful states, not just a bare `Default` export. The folder name matches the exported component's name, not the source filename, when they differ (e.g. `sonner.tsx` exports `Toaster`, so its story lives at `src/stories/components/Toaster/Toaster.stories.tsx`).

### `src/contexts/` — Context providers

`<Name>Context/` folder per context, split into `context.ts` (the `createContext` call + its type), `provider.tsx` (the `<Name>Provider` component), `consumer.ts` (the `use<Name>` hook), `index.ts` (barrel). Three so far: `src/contexts/ForgotPasswordContext/` (3-step forgot-password state), `src/contexts/RegistrationContext/` (5-step registration wizard state — org name/GST from step 1, email from step 2, `registrationToken` from step 3, mobile number from step 4), and `src/contexts/SessionContext/` (the logged-in user + their active organization for the whole `(private)` area). Each wizard step that depends on an earlier step's state guards itself with a `useEffect` that redirects back to step 1 if that field is empty (see any `register/*/page.tsx` or `forgot-password/*/page.tsx`) — a full page reload loses this in-memory state by design, it isn't persisted anywhere. Employee Invitation (`008`) doesn't have an entry here — it was originally a multi-route wizard with its own `EmployeeInviteContext`, but that was deleted once the screen collapsed into a single page with plain local `useState` (see `src/app/` above); don't recreate a context for it without a genuine cross-route need reappearing.

`SessionContext` is a different shape from the other two: its `provider.tsx` doesn't just hold state, it owns the `getMe()` fetch (loading/error/401-redirect) that every other private page used to duplicate, and renders the shared `Header` once loaded — so `(private)/layout.tsx` is just `<SessionProvider>{children}</SessionProvider>`. `useSession()` exposes `{ user, organization, isOwner }`. There used to also be a `setOrganization` setter for `company-settings/organizations/page.tsx` (a switch-active-organization screen) — both are gone now: the backend's `Employee` (the login entity, per `backend/CLAUDE.md`'s User → Employee merge) is permanently scoped to exactly one organization, so there's nothing left to switch between. `organization` itself still comes back from `getMe()` and is still rendered (in `header.tsx`, `dashboard/page.tsx`) — only the ability to change which one is active was removed.

### `src/types/` — Types

One file per domain: `<Domain>.type.ts`. `src/types/auth.type.ts` holds `AuthUser`, `src/types/organization.type.ts` holds `Organization` (no more `OrganizationMembership` — that was `GET /api/organizations/mine`'s shape, and that endpoint no longer exists per the User → Employee merge above). `src/types/employee.type.ts` holds `EmployeeTitle`/`EmployeeGender`/`EmployeePickerOption`/`ApproverLevel` (no `ModuleAccessKey`/`MODULE_ACCESS_OPTIONS` — Module Access was removed from the Access & Approval section after initial implementation, at explicit request; see `008`'s Open Questions) plus, added for `009`, `EmployeeListItem`/`EmployeeStatusFilterValue`/`EmployeeSortBy`/`EmployeeBasicInfo`; `src/types/project.type.ts` and `src/types/airline.type.ts` hold the minimal `Project`/`Airline` shapes for Step 2/3's pickers. Don't define shared types inline in components/API files.

### `src/utils/` — Utilities

```
src/utils/
├── apiManager/
│   └── apiManager.ts       # fetch wrapper (apiCall), ApiError, credentials handling
├── constants/
│   ├── route.constant.ts   # ROUTES object — all app navigation paths
│   └── regex.constant.ts   # EMAIL_REGEX, PHONE_REGEX, OTP_REGEX, PASSWORD_REGEX
└── helpers/
    └── validation.helper.ts  # isEmail/isPhone/isValidOtp/isStrongPassword etc.
```
`src/lib/` only holds `utils.ts` (the `cn()` class-merge helper) — matches `zp-frontend`'s own "tiny shared utilities" definition for that folder exactly; everything else moved out of `lib/` and into the buckets above.

Note: `apiManager.ts` uses `fetch`, not Axios — no reason to add an Axios dependency just to match `zp-frontend`'s HTTP client choice; the module's role (base URL, credentials, error normalization) is what's mirrored, not the library.

### Storybook

Config in `.storybook/`. Uses `@storybook/react-vite` (plain React, not `@storybook/nextjs`) — this project's Next 16 + the only Storybook version with matching Next-16 support (10.x) require a Node engine this environment doesn't meet, and `@storybook/nextjs`'s webpack/SWC integration doesn't work with Next 16.2.10 regardless. `react-vite` is a deliberate, verified-working choice, not a placeholder — fine as long as a component doesn't import `next/*` APIs directly. Stories glob (`src/**/*.stories.tsx`) picks up every story regardless of location, but all stories live under `src/stories/components/` by convention (see above) — no story files exist inside `src/components/` itself. `.storybook/main.ts` also serves `public/` as a static dir; that folder has no real assets right now (the default `create-next-app` placeholders were removed as unused) but must still exist on disk — `public/.gitkeep` keeps it from disappearing, since git doesn't track empty directories and Storybook's build fails outright if the configured `staticDirs` path is missing.

**`header.tsx` has no story, unlike every other shared component**: it's the first component to actually hit the "future component needs Next-specific mocking" case flagged above — it uses `next/link` and `next/navigation`'s `usePathname`/`useRouter`, neither of which `react-vite` mocks out of the box. Adding a real story means either a routing-mock addon (e.g. `storybook-addon-remix-react-router`-style shims don't apply here; something Next-specific) or wrapping it in a fake router context — deliberately not done as a side effect of the header-navigation story, since it's Storybook infrastructure work, not a header change. Revisit this the next time a component needs the same mocking, so it's solved once for both instead of twice.

Icons: `@phosphor-icons/react` (use the `*Icon`-suffixed exports, e.g. `EyeIcon` — the unsuffixed names are deprecated). Do not add `lucide-react` — it was removed in favor of Phosphor.

Fonts: `Montserrat` loaded via `next/font/google` in `src/app/layout.tsx`, exposed as the `--font-sans` CSS variable (wired to Tailwind's `font-sans` in `globals.css`). `Geist_Mono` remains for monospace (`--font-geist-mono` → `--font-mono`). Add fonts through `next/font/google`, not a manual `<style>`/`@import` tag — it self-hosts and avoids layout shift.
