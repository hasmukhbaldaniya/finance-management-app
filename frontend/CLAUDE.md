@AGENTS.md

## Overview

Next.js (App Router, TypeScript, `src/` directory, import alias `@/*`), styled with Tailwind CSS 4 + shadcn/ui. Implements login/forgot-password (`user-stories/001-authentication.md`), the 5-step company registration wizard (`user-stories/002-organization-signup.md`), the post-login header navigation (`user-stories/003-header-navigation.md` — its switch-organization screen is no longer implemented, see `SessionContext` below for why), Grade Management (`user-stories/004-grade-management.md`), Department Management (`user-stories/005-department-management.md`), Roles & Privileges Management (`user-stories/006-roles-and-privileges-management.md`), the owner-only Associated Organizations network (`user-stories/007-associated-organizations-network.md`), the 4-step Invite Employee wizard, Story 1 of 3 in the Employee Management epic (`user-stories/008-employee-invitation.md`), the Employee Listing screen (Story 2, `user-stories/009-employee-listing.md`), the Bulk Invite Employees CSV/XLSX upload screen (Story 3, `user-stories/010-bulk-invite-employees.md`), the invited employee's own 4-step Onboarding flow under `(public)/onboarding/` (Story 4, added after the fact, `user-stories/011-employee-onboarding.md`), the logged-in employee's own self-service Profile screen — a separate epic from Employee Management (`user-stories/012-employee-profile.md`), Category Management — the Create Category wizard, My Categories listing, Edit/Duplicate, and version history (`user-stories/013-category-creation.md` through `016-category-version-history.md`, see "Category Management" below), the Zoho SalesIQ live-chat widget mounted app-wide across the authenticated area (`user-stories/017-zoho-salesiq-integration.md`, see "Zoho SalesIQ" below), and Trip Management — Create Trip and the "My Trips" listing/search/filter/delete (`user-stories/018-trip-creation.md`/`019-trip-listing.md`, see "Trip Management" below). Structure is modeled on `zp-frontend`'s conventions (see `../../nextjs/zp-frontend/CLAUDE.md`), scaled down to what this app actually has — folders aren't pre-created until there's real content for them.

## Project Structure

```
src/
├── apis/                # API layer — one subfolder per domain (currently: auth/)
├── app/                 # Next.js App Router routes, split into (public)/(private) groups
├── components/          # Feature/shared components + shadcn primitives (ui/)
├── contexts/            # React context providers, one <Name>Context/ folder each
├── hooks/               # Shared hooks (currently: useInfiniteScroll.ts)
├── stories/             # Storybook stories for every component, one <PascalCaseName>/ folder each
├── types/               # TypeScript types per domain
└── utils/               # constants/, helpers/, apiManager/
```
`src/hooks/` holds one hook so far: `useInfiniteScroll.ts` — created once Grade and Department Management both needed the identical "load the next page when the user scrolls to the bottom" behavior (an `IntersectionObserver` watching a sentinel `<div>`, no "Load more" button); Roles & Privileges reuses it too, unchanged. It reads `onLoadMore`/`hasMore`/`isLoading` via a ref updated in an effect (not during render — `react-hooks/refs` flags writing a ref synchronously in the render body) so the observer itself is created once per mount instead of being torn down and recreated on every fetch. Reuse this for every future paginated list (Employees next), don't re-implement it inline. No `src/icons/` yet — not created until there's a real shared icon component to put in it (see `zp-frontend/CLAUDE.md`'s own layer description for what it's for when that day comes).

### `src/apis/` — API layer

One subfolder per domain, one file per endpoint: `src/apis/auth/` (login/logout/me/forgot-password/registration endpoints — registration's endpoints are still `auth/*` since they're all under the backend's `/api/auth/registrations/*` namespace; `012` added `changePassword.api.ts` here, not to `employee/`, since `PATCH /api/auth/me/password` lives under `/auth` on the backend too), `src/apis/organization/` (GST-availability check only — no more `listMine.api.ts`/`src/apis/user/`, both removed along with `GET /api/organizations/mine`/`PATCH /api/users/me/active-organization` per the User → Employee merge, see `backend/CLAUDE.md`), `src/apis/grade/`, `src/apis/department/`, `src/apis/role/` (each: list/create/update/updateStatus/delete/listMembers, one file each, matching the backend's `/api/grades*`/`/api/departments*`/`/api/roles*` routes one-to-one — `department/` and `role/` are straight copies of `grade/` with the noun swapped; `role/create.api.ts`/`update.api.ts` additionally take a `privileges: PrivilegeKey[]` argument), `src/apis/associated-organization/` (`list.api.ts`/`updateStatus.api.ts` only — no create/update/delete, matching the backend's read-mostly `/api/associated-organizations*` surface), `src/apis/project/` (`list.api.ts`/`create.api.ts`, for Step 2's Projects picker + inline "+ New Project"), `src/apis/airline/` (`list.api.ts` only, for Step 3's Airline picker), `src/apis/employee/`, the largest domain — split across two stories: `008`'s wizard mutations (`create.api.ts`, `updateCompanyAccess.api.ts`, `addFfNumbers.api.ts`, `saveApprovals.api.ts`, `sendInvite.api.ts`) plus `listForPicker.api.ts` for the approver picker (`GET /api/employees/approvers` — moved off the bare `/employees` path once `009` needed that for its own listing, see `backend/CLAUDE.md`'s Employee Listing section); `009`'s own `list.api.ts` (the main `GET /api/employees`), `resend.api.ts`, `updateStatus.api.ts`, `getDetail.api.ts`, and `updateBasicInfo.api.ts` (the last two power the Edit page); `010`'s `downloadBulkTemplate.api.ts`/`uploadBulkImport.api.ts`/`confirmBulkImport.api.ts`/`downloadBulkImportErrors.api.ts` (the first and last use `apiManager.ts`'s `downloadFile()`, the second uses its `uploadFile()` — see below, neither returns/sends plain JSON like every other `.api.ts` file in this codebase does); `012`'s `getMyProfile.api.ts`/`updateMyProfile.api.ts`/`setMyMobile.api.ts`/`resendMyMobileOtp.api.ts`/`verifyMyMobileOtp.api.ts` (all scoped to `/employees/me*`, no `:id` param — kept in the `employee/` domain since they're still about the `Employee` record, just the caller's own); and `src/apis/employee-onboarding/` (`011`, one file per `/api/employee-onboarding/*` endpoint: `verifyToken.api.ts`, `setPassword.api.ts`, `updateProfile.api.ts`, `setMobile.api.ts` — the one `apiCall` direct call in the domain, since it's a `PUT` and `postJson` is POST-only, matching `auth/setRegistrationMobile.api.ts`'s same reason — `sendMobileOtp.api.ts`, `verifyMobileOtp.api.ts`, `complete.api.ts`). Every `.api.ts` file just calls `apiCall`/`postJson` from `src/utils/apiManager/apiManager.ts` and returns its typed response. Every domain folder has an `index.ts` barrel; import from the domain barrel (`@/apis/auth`, `@/apis/organization`, `@/apis/grade`, `@/apis/department`, `@/apis/role`, `@/apis/associated-organization`, `@/apis/employee`, `@/apis/employee-onboarding`, `@/apis/project`, `@/apis/airline`), not individual `.api.ts` files, outside the domain itself. No query-key layer yet — this project doesn't use a data-fetching/cache library (no React Query), so there's nothing to key. If one gets added later, add `src/apis/queryKeys/<domain>.queryKeys.ts` to match `zp-frontend`'s convention at that point.

**Server-driven search/sort/pagination screens (Grade/Department Management, and their siblings to come) reset loading/error state synchronously at the top of an effect before refetching** — the standard shape for every future org-scoped CRUD screen (`src/app/(private)/company-settings/grades/page.tsx` and `.../departments/page.tsx` are the reference implementations — near-identical, `departments/page.tsx` is a straight copy with the noun swapped). `eslint.config.mjs` turns off `react-hooks/set-state-in-effect` project-wide for exactly this reason: without a data-fetching library, "fetch on a dependency change, resetting loading/error first" is the correct pattern (React's own docs list data-fetching as a valid effect use), and the rule's concern is an extra render pass, not a bug. Don't silence it per-line instead of relying on this — the whole point of turning it off centrally is that every future CRUD screen hits the exact same shape.

### `src/app/` — Routes

Two route groups (groups don't affect the URL — `/login` and `/dashboard` are unchanged):
- `src/app/(public)/` — unauthenticated screens (`login/`, `forgot-password/` and its steps, `register/` and its steps, `onboarding/` and its steps). No route protection.
- `src/app/(private)/` — authenticated screens: `dashboard/`, `trips/`, `claims/`, `approvals/`, `finance/`, `reports/`, `help/`, `profile/`, and `company-settings/{employees,categories,roles-privileges,grades,departments,associated-organizations}/` (no more `organizations/` — its switch-active-organization screen was removed, see `SessionContext` above). `categories/` is no longer a placeholder — see "Category Management" below for its full route tree. `trips/` is no longer a placeholder either — see "Trip Management" below. `employees/page.tsx` is now `009`'s real Employee Listing (replacing the old `ComingSoon` + bridge "Invite Employee" button). `profile/page.tsx` is now `012`'s real View Profile screen (replacing its own `<ComingSoon title="View Profile" />`) — see below for its shape. `company-settings/grades/page.tsx`, `.../departments/page.tsx`, and `.../roles-privileges/page.tsx` are the reference implementations for every other org-scoped CRUD screen — see `src/components/grade/`, `src/components/department/`, and `src/components/role/` (`form-dialog.tsx`/`members-dialog.tsx`/`status-dialog.tsx` each) for the Add/Edit-with-Delete-inside, Members-list, and toggle-confirmation dialog patterns to reuse. `role/` additionally has `view-dialog.tsx` (read-only, for the two seeded default roles — Edit/Delete/toggle are all disabled for them, both in the UI and re-enforced server-side) and `privilege-checkbox-list.tsx` (the shared 8-item checkbox group both `form-dialog.tsx` and `view-dialog.tsx` render, backed by `src/utils/constants/privilege.constant.ts` — kept byte-identical to the backend's copy). **The status toggle always opens a confirmation dialog before changing anything** — `grade/status-dialog.tsx`/`department/status-dialog.tsx`/`role/status-dialog.tsx` — there is no optimistic/instant toggle anywhere in this codebase, even though [004](../user-stories/004-grade-management.md)'s story doc originally specced one; that decision was superseded after implementation (see that doc's Open Questions), and Department/Roles were both built with the confirmation dialog from the start.

  **`company-settings/employees/page.tsx`'s filter UI matches Associated Organizations' pattern exactly, at explicit request** (not the search-bar-plus-dropdown-menu shape built first): a funnel/X toggle button next to the title (not top-right of the card like Associated Organizations — positioned beside "Employee Management" instead) opens a per-column filter row inside the table's `<TableHeader>`, same as that page — debounced text `Input`s under Employee Name/Email/Contact Number (Employee Name's filter also matches `employeeCode` server-side, since Employee ID has no dedicated column to attach its own filter to), a single-select `SelectField` under Employee Status (`"All"` + Active/Suspended/Pending Invitation — single-select, mirroring Associated Organizations' Registration/Status dropdowns, even though the backend's `status` param still accepts comma-separated multi-value). Closing the filter row via the X clears every filter, not just its visibility, exactly like that page's `handleCloseFilters`. Unlike Associated Organizations (only 2 of 7 columns sortable), every column here is sortable per `009`'s own spec — matching that page's pattern refers to the filter *row's* look/interaction, not an instruction to also restrict sorting. **Grade is a listing column too, added after `009` shipped, at explicit request** — `employee-invite/page.tsx` and `employees/[id]/edit/page.tsx` already required and saved `gradeId` from day one (Step 2's Company Access section), it just wasn't being read back onto the listing; `SORTABLE_COLUMNS`/`EmployeeListItem`/the CSV export header all gained a `grade` entry alongside the existing `role`/`department` ones, no new filter (it has no dedicated search input, same as Role/Department).

  **The empty-results state renders inside `TableBody`, not by swapping out the whole `<Table>`** — a real bug, fixed after the fact: the first version's `employees.length === 0` check replaced the entire table (header and open filter row included) with a bare "No Employees Found" line, so narrowing a filter to zero matches made the filter inputs disappear mid-interaction. Now it's a single row spanning every column, inside `TableBody`, so the header/filter row are always present once loaded. `company-settings/associated-organizations/page.tsx` has this exact same latent bug (same copied pattern, `rows.length === 0` swapping the whole table) — not fixed, since it wasn't asked for, but flagged here as the same root cause if it ever comes up.

  **`company-settings/employees/page.tsx`'s Suspend/Activate toggle is the one deliberate exception to "always confirm"** — per [009](../user-stories/009-employee-listing.md)'s own Flow, only Suspend opens a confirmation dialog (`src/components/employee/status-dialog.tsx`); clicking Activate on a Suspended row applies immediately with no dialog, since re-enabling access is the lower-risk direction. This is intentionally asymmetric, not a bug or an inconsistency with Grade/Department/Role's always-confirm pattern — don't "fix" it to match those without being asked. The listing also **excludes the logged-in caller's own row** entirely (added at explicit request, not in `009`'s original doc) — an admin manages their own account elsewhere, not through this table.

  **`company-settings/employees/[id]/edit/page.tsx`** is the Listing's Edit action — [009](../user-stories/009-employee-listing.md) explicitly defers this screen's field-by-field spec ("reuses [008](../user-stories/008-employee-invitation.md)'s rules verbatim... left for implementation time"), so it's built as a near-duplicate of `employees/invite/page.tsx`: same `SectionCard`/`SelectField`/`StepNav` structure, same validation, but pre-filled from `GET /api/employees/:id` and every write call updates instead of creates — no terminal "Send Invite" step, since Resend Invitation is its own separate row action on the listing, not part of editing. Given how much these two pages share, if a third near-identical form ever appears, that's the signal to extract the shared section markup into components rather than copy-pasting a third time — two is still an acceptable duplication, per this codebase's general "don't abstract until it hurts" bias.

  **`company-settings/employees/bulk-invite/page.tsx`** (`010`) is the Listing's third action — a "Bulk Invite" button (`ROUTES.EMPLOYEE_BULK_INVITE`) sits between Export and Invite on `employees/page.tsx`. Unlike every other CRUD/wizard screen in this app, it doesn't hold form fields in `useState` — it holds a single `BulkUploadSummary | null` returned by the backend after `uploadBulkImport`. Selecting a file runs frontend checks (extension, size, non-empty, and "not the exact same file already uploaded this session" via a `useRef`-held `{name, size, lastModified}` signature — refreshing the page is the only way to re-submit an identical file, matching `010`'s own rule) before the file ever reaches `apiManager.ts`'s `uploadFile()`; the resulting summary (Total/Success/Failed/Updated/New Employees) renders as a row of stat cards, a **Download Error Report** button appears only if `failed > 0`, and **Invite** (disabled until `created + updated > 0`) calls `confirmBulkImport` and redirects to the Listing on success — mirroring how `register/mobile/page.tsx` and `onboarding/mobile/page.tsx` both discard their own completion call's response and just redirect. **Cancel** makes no API call at all, since nothing is persisted until Invite is clicked (the upload's `BulkUpload`/`BulkUploadError` audit rows are written server-side regardless, but no `Employee` rows are).

  **`company-settings/employees/invite/page.tsx`** is a single-page form, not a multi-route wizard and not a list+dialog CRUD screen. [008](../user-stories/008-employee-invitation.md) originally specced this as a 4-step wizard (separate routes per step, gated by Continue/Back, each step's fields locked until the previous step's API call succeeded) — that was built first, then explicitly superseded at the user's request: all 4 sections (Basic Information, Company Access, FF Numbers, Access & Approval) render simultaneously on one page via `SectionCard` (`src/components/employee-invite/`), every field in every section is editable at any time in any order, and nothing is saved until a single "Send Invite" button at the bottom. See [008](../user-stories/008-employee-invitation.md)'s Open Questions for the full writeup — the per-step API endpoints themselves (`create`/`company-access`/`ff-numbers`/`approvals`/`send-invite`) are unchanged, only the frontend navigation model changed. No `EmployeeInviteContext`/`layout.tsx`/`StepShell` exist anymore (deleted when this collapsed to one page) — all wizard state is plain `useState` local to the single page component, since there's no cross-route state to persist. A left-rail `StepNav` (`src/components/employee-invite/step-nav.tsx`, `md:` and up only) lists the 4 sections as numbered steps and scroll-jumps to each `SectionCard` (matched by `id`); it's purely a navigation aid with an `IntersectionObserver` highlighting the in-view section — it does NOT gate editing or submission (the single-page "everything editable at once" model is the whole point of the redesign), so don't add step-completion locking to it.

  On submit, the page runs the 5 API calls in sequence (create → company-access → ff-numbers, only if any row was filled → approvals → send-invite), tracking the created `employeeId` in local state so a failed retry after the employee already exists doesn't re-call `createEmployee` (which would 409 on the now-duplicate email). The `createEmployee` call has its own nested try/catch specifically to map its 409s to the right inline field (email/contact number/Employee ID) — don't widen that mapping to cover the later calls' errors, none of which are field-specific in this UI.

  **Exception**: `company-settings/associated-organizations/page.tsx`'s Status column is a colored pill (green "Active"/muted "Disabled"), not a `Switch`, but it's still clickable and still instant/optimistic (flip immediately, revert-with-toast on failure), *not* confirmation-gated — this is not an oversight, it's what [007](../user-stories/007-associated-organizations-network.md) actually specifies (written and unchanged since before the Grade/Department/Role confirmation-dialog pattern existed) and nobody's asked to bring it in line since. If asked to "make it consistent with the other toggles," that's the one-line fix (swap the pill for a `Switch` + `status-dialog.tsx` like the others) — but don't do it unprompted, it's a deliberate per-story difference, not a bug. This page is also the **first screen in the app gated by role** — it doesn't render its own auth check; it relies on the header simply not showing its nav link for non-owners (see `header.tsx` below) and the backend 403ing directly if a non-owner reaches the URL/API anyway.

  Its filter UI is a deliberate one-off, redesigned after initial implementation against approved reference screenshots (see [007](../user-stories/007-associated-organizations-network.md)'s Open Questions) — don't copy this pattern onto Grade/Department/Role without a similar explicit design reference: a funnel icon (top-right of the title, `FunnelIcon`/`XIcon` from `@phosphor-icons/react`) toggles a per-column filter row that appears directly inside the table's `<TableHeader>` as a second `<TableRow>` — four debounced text `Input`s (Organization Name, Employee Name, Email Address, Contact No.) plus two `SelectField` dropdowns (Registration, Status; single-value, "All" + the field's fixed options — see "Dates and selects are shadcn-backed" below; these were hand-styled natives before that sweep). Closing the filter row via the X **clears every filter**, it's not just a visibility toggle. Only two of the seven columns (Organization Name, Employee Name) are sortable — the rest render as plain, non-interactive headers, also per the reference design. The `MultiSelectFilter` component originally built for this page (multi-select checkboxes via `DropdownMenuCheckboxItem`) was deleted once the design moved to single-select dropdowns — if a future story needs a real multi-select filter, `DropdownMenuCheckboxItem`'s `closeOnClick: false` default is still the right primitive to reach for, just rebuild it rather than expecting it to still exist.

  **`(public)/onboarding/`** (`011`) is the invited employee's own accept-invite flow, reached by clicking the link in the invite email (`?token=...`) — structurally it mirrors `register/`'s multi-route wizard (one route per step, state threaded through a context, not `employee-invite/`'s single-page collapse), since like registration each step's API call must actually succeed before the next is reachable. `page.tsx` (the entry route, i.e. Step 1) reads `token` via `useSearchParams` — wrapped in a `<Suspense>` boundary in the same file, since Next 16 fails a production build with a "Missing Suspense boundary" error otherwise — calls `verifyOnboardingToken` on mount (loading/error/invalid-link states), then renders the disabled-email + password/confirm-password form; `profile/page.tsx` (Step 2, title/first/last name, pre-filled from Step 1's response but editable, same `SelectField` the invite wizard already uses for Title); `mobile/page.tsx` (Step 3, Skip-or-Verify, same shape as `register/mobile/page.tsx`); `verify-mobile/page.tsx` (Step 4, Skip-or-Verify-OTP, same shape as `register/verify-mobile/page.tsx`). `layout.tsx` wraps all four in `OnboardingProvider`. Steps 2–4 guard with a `useEffect` that redirects to `ROUTES.LOGIN` (not back to Step 1, unlike `register/*`'s own steps) if the context's `token` is empty — there's no sensible "start over" destination without the original email-bound token from the link. All 3 of this flow's exit points (Step 3's Skip, Step 4's Skip, Step 4's successful Verify) call the same `completeOnboarding`, whose response is discarded the same way `register/mobile/page.tsx` and `register/verify-mobile/page.tsx` already discard `completeRegistration`'s — the backend sets the httpOnly session cookie itself, so the frontend only needs to `router.push(ROUTES.DASHBOARD)` after.

  **`(private)/profile/page.tsx`** (`012`) is the logged-in employee's own account screen, fetched once via `getMyProfile()` — unlike every Employee Management screen (`008`–`011`), this one is a genuinely separate epic (self-service, not admin-managing-others) and reuses no wizard/context machinery: it's a single read-only display (`ProfileField`, a local, page-only component — not extracted to `src/components/`, since nothing else renders it) with two dialog-triggered actions, matching Grade/Department/Role's dialog-based edit pattern rather than the multi-route wizards. `src/components/profile/edit-profile-dialog.tsx` handles Title/First/Last Name/DOB/Gender/Employee ID (all self-editable — Email and Organization/Role/Department/Grade are not, see `backend/CLAUDE.md`'s Employee Profile section) plus Contact Number, which has its own two-mode flow inside the same dialog: changing the number and clicking Save calls `setMyMobile()` and switches the dialog into an inline OTP-verification view (same shape as `onboarding/verify-mobile/page.tsx`'s Verify/Resend), only calling the parent's `onSaved` once that OTP verifies — the other fields save immediately regardless. `src/components/profile/change-password-dialog.tsx` is a separate, simpler dialog (current/new/confirm password, calling `changePassword()` from the `auth` domain, not `employee`, matching where the backend endpoint lives). **A known, deliberately-uncorrected gap**: saving a name change here doesn't refresh `SessionContext`, so the header's cached name stays stale until the next full page load — `SessionContext` has no refetch/invalidation hook to call into yet (see its own section below), and adding one was out of scope for this story.

  Gated by `src/proxy.ts`'s `matcher`; any new authenticated route must be added under `(private)/` **and** to that matcher array. `(public)/onboarding/` is deliberately **not** added there — see `backend/CLAUDE.md`'s Employee Onboarding section for why this flow has no session at all until its final step. `(private)/layout.tsx` wraps every page in `SessionProvider` (see `src/contexts/`), which does the one `getMe()` fetch for the whole private area and renders the shared `Header` — individual pages must not re-fetch `getMe()` or render their own header/logout, they consume `useSession()` instead (see `dashboard/page.tsx` for the pattern). `useSession()` also exposes `isOwner` (from `GET /me`'s `isOwner` field) — the only place in the frontend that knows whether the current user owns their active organization; `Header` uses it to conditionally show the Associated Organizations (Network) sub-header link, and any future owner-gated UI should read it from here rather than re-deriving it.

Route paths are centralized in `src/utils/constants/route.constant.ts` (`ROUTES` object) — never hardcode a path in `href`, `router.push`/`replace`, or a `new URL(...)` call.

Route protection: `src/proxy.ts` (Next 16 renamed Middleware to Proxy; it defaults to the Node.js runtime, not Edge) verifies the session JWT itself via `jsonwebtoken` before allowing access to matched routes — it doesn't just check cookie presence. `AUTH_COOKIE_NAME` and `JWT_SECRET` env vars must match the backend's exactly (see `.env.example`); a mismatch fails safe (redirect to `/login`) but locks out every user if only one side is updated. `JWT_SECRET` has no fallback — the app throws at startup if it's unset, same as the backend — so an insecure default can't accidentally ship to production.

### `src/components/` — Components

- `src/components/ui/` — shadcn/ui primitives, copied (not npm-installed) via `npx shadcn add <component>` (or the `shadcn` MCP server in the root `.mcp.json`). Kebab-case filenames — this is shadcn's own convention, don't rename to fight the tool's update path. `dropdown-menu.tsx` is base-ui-backed (`@base-ui/react/menu`) like `button.tsx`/`input.tsx` — its subcomponents (`DropdownMenuItem`, `DropdownMenuTrigger`, etc.) render as a custom element via the `render` prop (base-ui's equivalent of Radix's `asChild`), e.g. `<DropdownMenuItem render={<Link href={...}>Label</Link>} />`, not `asChild` + a child element. Its icons were swapped from the shadcn CLI's default `lucide-react` to `@phosphor-icons/react` after adding it, matching this project's icon convention below — always check a freshly-added shadcn file for stray `lucide-react` imports. **`DropdownMenuLabel` must be a descendant of `DropdownMenuGroup`** — base-ui's underlying `Menu.GroupLabel` throws `"Menu group parts must be used within <Menu.Group>"` if rendered standalone; wrap it (and the items it labels) in a `DropdownMenuGroup`, or use a plain styled `<div>` instead if the text isn't actually labeling a group of items below it (see `header.tsx`'s Profile menu for the latter). `table.tsx`/`dialog.tsx`/`switch.tsx` (added for Grade Management) and `checkbox.tsx` (added for Roles & Privileges, its own privilege checklist) are the same base-ui-backed style — `dialog.tsx` and `checkbox.tsx` both needed the same `lucide-react` → `@phosphor-icons/react` icon swap as `dropdown-menu.tsx` did; check for it every time `npx shadcn add` pulls in a new primitive, it isn't a one-off.
- Truly cross-feature components (`auth-card.tsx`, `password-input.tsx`, `logo.tsx`, `header.tsx`, `coming-soon.tsx`) stay flat in `src/components/` — kebab-case filenames, not `zp-frontend`'s `PascalCase/<Name>.component.tsx`. `header.tsx` is the one shared component rendered outside `(public)`/`(private)` page bodies — it's mounted once by `(private)/layout.tsx`'s `SessionProvider`, not by individual pages. Two distinct nav patterns live in it, chosen per what the item represents: **Company Settings** is a plain `Link` (not a `DropdownMenu`) whose destinations render as a second nav row — a sub-header — directly under the main header whenever `pathname.startsWith("/company-settings")`, since they're peer pages a user switches between, not a transient action list. **Profile** stays a `DropdownMenu` popup, since its contents (identity display, View Profile, Logout) are a one-off action set, not pages to navigate between. Don't reach for `DropdownMenu` by default for every "item with children" — pick the sub-header pattern when the children are sibling routes.
- **Components unique to one feature get their own `<feature>/` folder** — `src/components/grade/`, `src/components/department/` (each: `form-dialog,members-dialog,status-dialog`), `src/components/role/` (those three plus `view-dialog.tsx` and `privilege-checkbox-list.tsx`), `src/components/employee-invite/` (`section-card.tsx` — a title/description + bordered-card wrapper, one per section on the single-page Invite Employee form (see `src/app/` above); `step-nav.tsx` — the left-rail scroll-jump nav — `native-select.tsx` used to live here too, deleted once `SelectField` replaced every `NativeSelect` call site, see "Dates and selects are shadcn-backed" below), and `src/components/employee/` (`status-dialog.tsx` — the Suspend confirmation dialog only; unlike `grade/department/role`'s `status-dialog.tsx`, this one is never opened for the Activate direction, see `src/app/` above) — imported as `GradeFormDialog` from `@/components/grade/form-dialog`, `SectionCard` from `@/components/employee-invite/section-card`, `EmployeeStatusDialog` from `@/components/employee/status-dialog`, etc. — the feature name lives in the folder, not repeated in the filename. This revisits the flat-components decision above: it made sense with one feature area (auth) and a handful of files, but stopped making sense once Grade/Department/Roles each grew several tightly-coupled dialog components. Follow this shape whenever a feature needs more than one component of its own, not something to defer indefinitely.

- **Dates and selects are shadcn-backed, not native, as of a whole-app sweep** — `ui/calendar.tsx`, `ui/popover.tsx`, `ui/select.tsx`, `ui/textarea.tsx` were added via `npx shadcn add` (icons swapped from the CLI's default `lucide-react` to `@phosphor-icons/react`, same as every other base-ui-backed primitive). On top of those, three cross-feature composed components live flat in `src/components/` (not `ui/`, since they're this app's own composition, not raw shadcn output): `date-picker.tsx` (`DatePicker`, a Popover+Calendar single-date picker) and `date-time-picker.tsx` (`DateTimePicker`, the same plus a time `Input` inside the popover) both operate on the exact string shapes native `<input type="date">`/`type="datetime-local">` already used (`"YYYY-MM-DD"`/`"YYYY-MM-DDTHH:mm"`, `onChange` receiving the string directly rather than an event) — swapping one in only ever means changing the component, never the surrounding state's type. `select-field.tsx` (`SelectField`) replaces native `<select>` the same way (plain string value/`onValueChange`, `""` still valid for an "All"-style option) — internally it maps `""` to a private sentinel since Base UI's `Select`, like Radix's, reserves the empty string to mean "nothing selected," but callers never see that translation. A native `<input type="time">` on its own (no date) is the one exception left alone — no shadcn time-only picker pattern exists, and `expense-field-renderer.tsx`'s `time` `CategoryFieldType` case still uses it directly.

  **The one real exception is a genuine multi-select** (`category/policy-eligibility-section.tsx`'s eligibility-entity picker) — shadcn's `Select` is single-value only, so there's no drop-in swap for a `<select multiple>`. That one instead uses this codebase's own already-established multi-select shape (`DropdownMenu` + `DropdownMenuCheckboxItem` with `closeOnClick={false}`, the same pattern `ziptrrip-category-picker.tsx` already used) — reach for that shape again for any future real multi-select, `SelectField` isn't it.
- Every component's story — `ui/` primitives included — lives under `src/stories/components/<PascalCaseName>/<PascalCaseName>.stories.tsx` (e.g. `src/stories/components/Button/Button.stories.tsx`, `src/stories/components/AuthCard/AuthCard.stories.tsx`), one directory per component, importing the component via its `@/components/...` path rather than a relative import — never colocated next to the component file itself, even for shadcn-managed `ui/` files. Required for every genuinely shared/reusable component (not one-off page-specific markup), covering its meaningful states, not just a bare `Default` export. The folder name matches the exported component's name, not the source filename, when they differ (e.g. `sonner.tsx` exports `Toaster`, so its story lives at `src/stories/components/Toaster/Toaster.stories.tsx`).

### `src/contexts/` — Context providers

`<Name>Context/` folder per context, split into `context.ts` (the `createContext` call + its type), `provider.tsx` (the `<Name>Provider` component), `consumer.ts` (the `use<Name>` hook), `index.ts` (barrel). Four so far: `src/contexts/ForgotPasswordContext/` (3-step forgot-password state), `src/contexts/RegistrationContext/` (5-step registration wizard state — org name/GST from step 1, email from step 2, `registrationToken` from step 3, mobile number from step 4), `src/contexts/OnboardingContext/` (`011`'s 4-step onboarding wizard state — `token`/`email`/`title`/`firstName`/`lastName` from Step 1's verify-token response, `countryCode`/`contactNumber` from Step 3 — a straight structural copy of `RegistrationContext`, right down to the `reset()` shape, since it's the same "multi-route wizard, state threaded through steps via context" problem), and `src/contexts/SessionContext/` (the logged-in user + their active organization for the whole `(private)` area). Each wizard step that depends on an earlier step's state guards itself with a `useEffect` that redirects back to step 1 if that field is empty (see any `register/*/page.tsx` or `forgot-password/*/page.tsx`) — except `OnboardingContext`'s own steps 2–4, which redirect to `ROUTES.LOGIN` instead, see `src/app/` above for why. A full page reload loses this in-memory state by design, it isn't persisted anywhere. Employee Invitation (`008`) doesn't have an entry here — it was originally a multi-route wizard with its own `EmployeeInviteContext`, but that was deleted once the screen collapsed into a single page with plain local `useState` (see `src/app/` above); don't recreate a context for it without a genuine cross-route need reappearing.

`SessionContext` is a different shape from the other two: its `provider.tsx` doesn't just hold state, it owns the `getMe()` fetch (loading/error/401-redirect) that every other private page used to duplicate, and renders the shared `Header` once loaded — so `(private)/layout.tsx` is just `<SessionProvider>{children}</SessionProvider>`. `useSession()` exposes `{ user, organization, isOwner, isSalesIqAvailable }` (the last one added for `017`, see "Zoho SalesIQ" below). There used to also be a `setOrganization` setter for `company-settings/organizations/page.tsx` (a switch-active-organization screen) — both are gone now: the backend's `Employee` (the login entity, per `backend/CLAUDE.md`'s User → Employee merge) is permanently scoped to exactly one organization, so there's nothing left to switch between. `organization` itself still comes back from `getMe()` and is still rendered (in `header.tsx`, `dashboard/page.tsx`) — only the ability to change which one is active was removed.

### Zoho SalesIQ (017) — frontend-only, degrades silently

`src/components/zoho-salesiq-widget.tsx` mounts alongside `Header` inside `SessionProvider` (`src/contexts/SessionContext/provider.tsx`) — the one place `user`/`organization` are already guaranteed non-null by the time it renders, so the widget never has to handle an "identity not yet loaded" race. It renders nothing itself if `NEXT_PUBLIC_ZOHO_SALESIQ_WIDGET_URL` is unset (see `.env.example`) — no script tag, no error, the app behaves exactly as if this feature didn't exist. **The env var holds the full loader-script URL** (`https://salesiq.zohopublic.in/widget?wc=<widget-code>` for this account), not a bare widget code — confirmed against a real account's actual embed snippet during implementation, correcting an initial assumption that the domain was a fixed `salesiq.zohopublic.com` and that the widget code was set via a `widgetcode` JS property; neither is true — the domain is region-specific (`.in` here) and the code is a query param on the URL itself, read by the loader script directly off its own `src`, not from any JS object this file controls. When configured, the component follows Zoho's own canonical embed shape via two `next/script` tags in order: an inline one (`strategy="afterInteractive"`) that sets `window.$zoho.salesiq = { ready: () => {...} }` (matching Zoho's real snippet exactly, nothing invented) — visitor name/email/organization are JSON-encoded before being embedded in that inline script's text, so they're safe JS string literals regardless of content, not raw string concatenation — followed by the external loader script (`src={WIDGET_SCRIPT_URL}`), whose `onLoad`/`onError` flip `SessionContext`'s `isSalesIqAvailable` flag. `help/page.tsx`'s "Chat with Us" button (replacing that page's old `<ComingSoon title="Help" />`) reads `isSalesIqAvailable` and hides itself entirely rather than rendering a button that would no-op — the same "degrade gracefully, never surface an error" posture the story doc specifies throughout. `openZohoSalesIqChat()` (exported from the same file) opens the floating chat panel programmatically via `window.$zoho.salesiq.floatwindow.visible("show")`, wrapped in optional chaining + try/catch since Zoho's own JS API surface is versioned and not something this codebase controls — see `017`'s Open Questions for why exact method names weren't more rigidly assumed (the loader URL shape, unlike those method names, is now confirmed rather than assumed). Only `NEXT_PUBLIC_` client-side config exists for this feature; there is no backend involvement at all (checked `backend/src/app.ts`'s `helmet()` CSP config on the assumption it might need a `script-src` allowlist entry for Zoho's domain — it doesn't, since that CSP only governs the backend's own JSON API responses, not pages the Next.js frontend serves). The real widget URL lives only in `frontend/.env.local` (gitignored) — never committed, `.env.example` only ever holds a blank placeholder.

### `src/types/` — Types

One file per domain: `<Domain>.type.ts`. `src/types/auth.type.ts` holds `AuthUser`, `src/types/organization.type.ts` holds `Organization` (no more `OrganizationMembership` — that was `GET /api/organizations/mine`'s shape, and that endpoint no longer exists per the User → Employee merge above). `src/types/employee.type.ts` holds `EmployeeTitle`/`EmployeeGender`/`EmployeePickerOption`/`ApproverLevel` (no `ModuleAccessKey`/`MODULE_ACCESS_OPTIONS` — Module Access was removed from the Access & Approval section after initial implementation, at explicit request; see `008`'s Open Questions) plus, added for `009`, `EmployeeListItem`/`EmployeeStatusFilterValue`/`EmployeeSortBy`/`EmployeeBasicInfo`; `src/types/project.type.ts` and `src/types/airline.type.ts` hold the minimal `Project`/`Airline` shapes for Step 2/3's pickers; `src/types/category.type.ts` holds the full Category Management type tree (see "Category Management" below) — `CategorySnapshot`, `CategoryField`, `CategoryPolicy`, etc., mirroring the backend's own `CategorySnapshot` shape field-for-field. `src/types/trip.type.ts`/`country.type.ts`/`city.type.ts` hold Trip Management's shapes (see "Trip Management" below). Don't define shared types inline in components/API files.

### Category Management (013-016) — the largest feature in this codebase

`src/apis/category/` (one file per backend endpoint, same one-file-per-endpoint convention as every other domain) and `src/types/category.type.ts` mirror `backend/src/utils/category-snapshot.ts`'s `CategorySnapshot` shape field-for-field, so `GET /api/categories/:id`'s response can be dropped straight into `CategoryWizardContext` with no transformation layer. `src/utils/constants/category.constant.ts` is kept byte-identical to the backend's own copy (field types, Ziptrrip categories, list value sources, city list, count/length limits) — the same "kept in sync by hand" convention `privilege.constant.ts` already established for Roles & Privileges.

**Route tree** (`src/app/(private)/company-settings/categories/`):
```
layout.tsx                     — wraps everything below in CategoryWizardProvider
page.tsx                       — 014's My Categories grid
new/page.tsx                   — Step 1, brand-new create OR duplicate (reads ?duplicateFrom=<sourceId>)
[id]/page.tsx                  — 016's read-only Category Details (reads ?version=X.Y, defaults to latest)
[id]/basic-details/page.tsx    — Step 1, editing an existing (draft or active) category
[id]/expense-form/page.tsx     — Step 2
[id]/policies/page.tsx         — Step 3
[id]/project-policies/page.tsx — Step 4 (terminal — always ends in activation, see below)
```
`new/` and `[id]/*` are siblings, not nested, but both need the same in-memory wizard state to survive the client-side navigation from `new` (no id yet) into `[id]/expense-form` (id just returned by Step 1's `POST`) — solved by putting `CategoryWizardProvider` at the shared parent `layout.tsx` rather than inside `[id]/layout.tsx`, so it never unmounts across any of these routes. This is a different shape from `RegistrationContext`/`OnboardingContext` (both single-flow, one context per multi-route wizard) — `CategoryWizardContext` additionally has to support three distinct entry modes (create, edit, duplicate) sharing the same routes and steps, not three separate route trees.

**Every `/[id]/<step>/page.tsx` always refetches via `useLoadCategory` (`src/components/category/use-load-category.ts`) on mount**, rather than trusting whatever's already in context — matching `008` Edit page's own always-refetch precedent. This is deliberately simple over clever: one extra `GET` per step visit, in exchange for never showing stale data after a deep link from the listing's Edit icon or a browser back/forward navigation.

**`WizardStepNav` (`src/components/category/wizard-step-nav.tsx`) is a gated, route-based step nav — a different component from `employee-invite/step-nav.tsx`'s scroll-jump variant**, which never gates anything since that form collapsed to a single page (see that section above). Here, a step is only clickable once `highestStepIndexReached` (tracked in `CategoryWizardContext`, seeded from how much data already exists when an in-progress category loads — e.g. a category with saved fields but no policies yet seeds `highestStepIndexReached = 1`) is at or past it, matching `013`'s own "completed steps stay clickable, future steps are greyed out" spec.

**A brand-new field's Conditional Visibility can target another brand-new field added in the same sitting** — `013`'s own Flow treats "a Dropdown field exists in the form" as local UI state, not persisted state, so two fields added together in Step 2 before ever saving need to be able to reference each other despite neither having a real database id yet. `expense-form-builder.tsx` assigns a negative, request-scoped sentinel as a new field's `id` (via a decrementing `useRef` counter) instead of leaving it `undefined`; `CategoryFieldDraft`'s own doc comment in `category.type.ts` explains this further. The backend resolves these sentinels to real ids once every field in the batch is persisted (see `backend/CLAUDE.md`'s Category Management section) — after every save, the frontend re-fetches via `getCategoryDetail` and replaces its local field list wholesale, so temp ids never survive past a single save.

**`PolicyCard` (`src/components/category/policy-card.tsx`) is one shared component behind Claim, Exception, and Project Policies** — parameterized by a `PolicyKind` (`"claim" | "exception" | "project"`) that drives which eligibility types `policy-eligibility-section.tsx` offers and how many Rule levels `policy-rules-section.tsx` permits, mirroring the backend's own `PolicyValidationOptions` split for the identical reason (013 confirms Project Policies are "identical shape to an Exception Policy"). `policy-approval-flows-section.tsx` + `approval-level-editor.tsx` render the always-present Default Flow plus up to 5 numbered Levels; a stage's AND/OR approvers are edited as a flat list of "slots" (`{logicGroup, employeeId}`) rather than the nested `approverGroups` shape the API actually uses — `+OR` appends a slot sharing the previous slot's `logicGroup`, `+AND` appends one with a fresh `logicGroup`, and the flat list is regrouped into `approverGroups` on every change. `policy-factory.ts` holds `createBlankPolicy`/`duplicatePolicy`, used identically by Step 3 (claim/exception) and Step 4 (project).

**Step 4's Save as Draft and Submit call the exact same handler** — per `013`'s API Design, `PUT .../project-policies` is confirmed as the one terminal call with no separate activate endpoint, so both buttons in `project-policies-form.tsx` invoke identical logic; the only UI difference is that Save as Draft is hidden once `status === "active"` (same as every other step), not a behavioral one.

**Known, deliberate scope simplifications** (functionally complete against every 013/014/015/016 acceptance criterion, but not pixel-matched to the reference screenshots' exact interaction polish):
- Reordering in `field-list.tsx` is up/down buttons, not a drag handle — same end state (a reorderable list persisted via `position`) without a drag-and-drop library dependency.
- The Formula Builder and Red Flag "Formula Based" inputs are plain text fields with a helper line listing available field names, not a live `{{`-triggered autocomplete popover.
- `version-history-drawer.tsx` reuses the existing centered `Dialog` primitive rather than a true right-side sliding drawer — this codebase has no drawer/sheet component yet.
- The Category Details page's field/policy rows (`field-summary.tsx`, `policy-summary.tsx`) render configuration as compact label/value text (including a raw JSON dump for a field's type-specific `config` blob) rather than a fully bespoke per-field-type layout.
- Policy Rules/Approval Flows sections are collapsible per policy card, not independently collapsible within a card as `013` describes.
These are flagged here so a future pass can pick any of them up deliberately, not because they were missed.

### Trip Management (018-019) — no owner gate, a searchable server-backed city picker, and a genuinely open status question

`src/app/(private)/trips/page.tsx` (`019`'s "My Trips" listing, replacing the old `<ComingSoon title="Trips" />`) and `trips/new/page.tsx` (`018`'s Create Trip) are this codebase's first employee-facing screens with no owner/Company-Administrator gate anywhere in their data fetching — every API call just relies on the backend scoping by the logged-in employee's own id, matching `018`'s explicit "personal record, not admin screen" posture (see `backend/CLAUDE.md`'s own Trip Management section). There's no `CategoryWizardContext`-style shared context here — Create Trip is a single-screen, one-shot-submit form (`create-trip-form.tsx`), not a multi-step wizard, so plain local `useState` is enough.

**`src/components/trip/city-select.tsx` is this codebase's first *server-backed* searchable dropdown** — every earlier searchable picker (`ziptrrip-category-picker.tsx`, Category Management's eligibility/options pickers) filters a small list already held in memory; a worldwide city catalog is too large for that, so this one debounces (250ms) and calls `GET /api/cities?search=` on every keystroke, backed by `@phosphor-icons/react`'s `DropdownMenu` + `Input` combo the same way `ziptrrip-category-picker.tsx` already established, just swapping a static array for a live fetch.

**Currency is Indian Rupees with Indian digit grouping, confirmed by `019`'s reference screenshot** — `src/utils/helpers/format.helper.ts`'s `formatInr()` uses `Intl.NumberFormat("en-IN", ...)` specifically (not `"en-US"`), so `150000` renders as `1,50,000.00`, not `150,000.00`. This is the first money-formatting helper in this codebase; `amount-chip.tsx` (the ₹-icon chip rendering Total Amount/Approved Amount on each trip row) is its only consumer so far.

**`Trip.status` has four possible values, but this codebase's own Create Trip flow only ever produces `"new"`.** `trip.constant.ts`'s `TRIP_STATUS_OPTIONS`/`TRIP_STATUS_BADGE_STYLE`/`TRIP_STATUS_LABEL` cover all four (`draft`/`new`/`pending_for_approval`/`approved_for_reimbursement`, styled per `019`'s own Status Values & Display table) because the listing has to be able to display and filter by whatever's actually in the database — but nothing in this codebase's UI has a Save-as-Draft action, so a `"Draft"`-status trip (with its delete affordance) is a state the listing can *render* but this implementation can never *produce*. This is `018`'s own flagged, unresolved gap, not a bug — don't add a Save-as-Draft button to `create-trip-form.tsx` without re-confirming that's actually wanted, per `018`'s Open Questions.

**"Create Claim" renders disabled on the Trips list** (`trips/page.tsx`) rather than being omitted or silently doing nothing on click — `019`'s Out of Scope explicitly says Create Claim itself isn't specified anywhere yet, and a present-but-inert button communicates "coming soon" better than either removing it (which would contradict the reference screenshot) or wiring an onClick that does nothing.

The Trip Start Date/Created Date filters in `trip-filters.tsx` are `DatePicker`s treated as **exact-calendar-day** matches (matching the backend's own `dayRange()` resolution) — `019`'s Open Questions flags this as an assumption, not a confirmed final design, since a single-day filter is a narrow way to slice a trip list and a date-range picker might actually be wanted instead.

### Storybook

Config in `.storybook/`. Uses `@storybook/react-vite` (plain React, not `@storybook/nextjs`) — this project's Next 16 + the only Storybook version with matching Next-16 support (10.x) require a Node engine this environment doesn't meet, and `@storybook/nextjs`'s webpack/SWC integration doesn't work with Next 16.2.10 regardless. `react-vite` is a deliberate, verified-working choice, not a placeholder — fine as long as a component doesn't import `next/*` APIs directly. Stories glob (`src/**/*.stories.tsx`) picks up every story regardless of location, but all stories live under `src/stories/components/` by convention (see above) — no story files exist inside `src/components/` itself. `.storybook/main.ts` also serves `public/` as a static dir; that folder has no real assets right now (the default `create-next-app` placeholders were removed as unused) but must still exist on disk — `public/.gitkeep` keeps it from disappearing, since git doesn't track empty directories and Storybook's build fails outright if the configured `staticDirs` path is missing.

**`header.tsx` has no story, unlike every other shared component**: it's the first component to actually hit the "future component needs Next-specific mocking" case flagged above — it uses `next/link` and `next/navigation`'s `usePathname`/`useRouter`, neither of which `react-vite` mocks out of the box. Adding a real story means either a routing-mock addon (e.g. `storybook-addon-remix-react-router`-style shims don't apply here; something Next-specific) or wrapping it in a fake router context — deliberately not done as a side effect of the header-navigation story, since it's Storybook infrastructure work, not a header change. Revisit this the next time a component needs the same mocking, so it's solved once for both instead of twice.

Icons: `@phosphor-icons/react` (use the `*Icon`-suffixed exports, e.g. `EyeIcon` — the unsuffixed names are deprecated). Do not add `lucide-react` — it was removed in favor of Phosphor.

Fonts: `Montserrat` loaded via `next/font/google` in `src/app/layout.tsx`, exposed as the `--font-sans` CSS variable (wired to Tailwind's `font-sans` in `globals.css`). `Geist_Mono` remains for monospace (`--font-geist-mono` → `--font-mono`). Add fonts through `next/font/google`, not a manual `<style>`/`@import` tag — it self-hosts and avoids layout shift.
