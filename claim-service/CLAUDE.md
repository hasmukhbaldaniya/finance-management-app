# CLAUDE.md

This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

## Overview

Node.js + Express 5 + TypeScript backend (`strict: true`, `moduleResolution: nodenext`), ORM Sequelize over its own PostgreSQL database (`pg`/`pg-hstore`). **This is `backend/`, renamed in place per `docs/PLANS/microservices-plan.md`'s Phase 4** — auth/registration, Employee Management, Employee Profile, and Grade/Department/Role/Associated-Organizations/Project/Airline had already moved to `auth-service` in Phase 3; this final rename+re-database is what turned the remaining Category/Trip/Claim/Split-Claim code into a real standalone service with its own Postgres database (`claim_service`, port `5434`, a separate container from both `backend`'s original DB and `auth-service`'s own), instead of a monolith directory that just happened to have less in it. Covers Category Management (`013`-`016`), Trip Management (`018`-`019`), Claim Management (`022`-`024`), and Split Claim (`025`) — see each section below. No service/repository layer beyond `src/services/` (thin HTTP clients for the three standalone microservices this calls — `ai-extraction.service.ts` for `ai-service`, `communications.service.ts` for `communications-service`, `auth.service.ts` for `auth-service`), controllers talk to models directly.

Models: `Category`/`CategoryZiptrripMapping`/`CategoryField`/`CategoryPolicy`/`CategoryPolicyEligibility`/`CategoryPolicyRule`/`CategoryApprovalLevel`/`CategoryApprovalStage`/`CategoryApprovalStageApprover`/`CategoryVersion` (Category Management's 10 tables — see "Category Management" below), `Country`/`City`/`Trip` (Trip Management — see "Trip Management" below), `Claim`/`Expense`/`ClaimInvoiceFile` (Claim Management — see "Claim Management" below), `ExpenseSplitRequest`/`ExpenseSplitRequestMember` (Split Claim — see below). `Employee`/`Organization`/`Grade`/`Department`/`Role`/etc. are **not** models here (moved to `auth-service` in Phase 3) — see "Cross-service reads" below. **`AiExtractionLog` is not a model here either** — Phase 4 moved it sideways into `ai-service`'s own MySQL database (see "Claim Management" below for why and what changed as a result); this service now only holds `ClaimInvoiceFile`/`Expense`/etc., not the audit trail for what was done to them.

**Real data migration, not a greenfield service**: this service's Postgres database was seeded via `pg_dump`/restore of the original monolith's 18 claim-domain tables (schema via replaying the 9 relevant historical migration files verbatim — with cross-service FK constraints into `organizations`/`employees` stripped out first, see "Cross-service reads" below — data via `--data-only` dump/restore). Row counts were verified to match exactly across every table, and the on-disk `uploads/claims/` directory (real uploaded invoice files) moved with the directory rename automatically.

**Email delivery moved to the standalone `communications-service` microservice** (Phase 1) — `src/utils/split-request-mailer.ts` (the one remaining mailer here, for Split Claim notifications) builds the same subject/body text it always did and calls `src/services/communications.service.ts` (POSTs to that service's `/api/notifications/email`) instead of holding its own `nodemailer` transporter. This service no longer depends on `nodemailer` at all.

## Common Commands

npm run migrate               # sequelize-cli db:migrate

## Architecture

Structure: `src/{config,models,controllers,routes,middleware,migrations,seeders,services}`.

### The 9 migrations that define this schema had their cross-service FK constraints stripped

`src/migrations/` holds only the 9 claim-domain migration files (everything from
`20260708190000-create-category-management-tables.js` onward) — the 19 auth-domain migrations that
used to sit alongside them (historical record from before Phase 3) were deleted once `auth-service`
confirmed its own authoritative copy; there was no reason for two copies of the same 19 files once
this service got its own dedicated, empty database to build fresh. Several of the 9 remaining files
had **real, literal `references: { model: "organizations"/"employees", key: "id" }` FK constraints**
that don't work anymore — a foreign key can't span two separate Postgres databases — so those were
edited to plain, unenforced integer columns before being replayed (each edit is commented in place
in the migration file itself). Same-service references (`categories`, `claims`, `expenses`, `trips`,
etc.) were left untouched. **A real, accepted consequence**: several of these columns used to
`onDelete: CASCADE`/`SET NULL` off `Employee`/`Organization` — a deleted Employee in `auth-service`
no longer cascade-deletes their `Trip`/`Claim`/`ExpenseSplitRequest*` rows here, the same category of
gap already documented for `AiExtractionLog`'s own cross-service FKs (see "Claim Management" below).

`20260710120000-seed-default-categories.js` (backfills every organization with 4 sample categories)
had a subtler version of the same problem — it queries `organizations` directly to find every org to
backfill. Since that table is now a genuine cross-service dependency for what's otherwise a one-time
migration, it opens a second, throwaway `pg` connection straight to `auth-service`'s database
(env-configurable, `AUTH_DB_HOST`/`AUTH_DB_PORT`/etc., defaulting to the local docker-compose
`postgres-auth` service) just to read organization ids, then writes to this service's own
`categories`/`category_fields` tables normally. This is a one-off migration-time read — nothing else
in this codebase opens a direct connection to another service's database.

### Cross-service reads — this backend has no local Employee/Organization/Grade/Role/Airline data anymore

Per Phase 3 of `docs/PLANS/microservices-plan.md`, `Employee` and every other identity/org table now
live in `auth-service`'s own database. Two things changed as a result:

- **`req.organizationId` comes straight off the verified JWT claim, with zero DB/HTTP calls.**
  `middleware/require-auth.ts` reads it directly from the access token (`utils/jwt.ts`'s
  `AccessTokenPayload`), which `auth-service` embeds at issue time — safe because
  `Employee.organizationId` is immutable once set. This replaced a `getActiveOrganizationId(userId)`
  helper (`utils/auth.ts`, since deleted entirely — nothing here needs it anymore) that used to do an
  `Employee.findByPk` on nearly every request; every controller that used to `await
  getActiveOrganizationId(req.userId)` (`category*`, `claim`, `trip`, `split-request` controllers)
  now just reads `req.organizationId`.
- **Real employee names/emails and the `Airline` catalog come from `auth-service`'s internal API**,
  via `src/services/auth.service.ts` (a thin HTTP client, same shape as `ai-extraction.service.ts`/
  `communications.service.ts`): `lookupEmployees(ids)` (→ `POST /api/internal/employees/lookup`) for
  `category.controller.ts`'s "created by" display, `split-request.controller.ts`'s colleague
  validation/requester/member names, and `duplicate-expense-check.ts`'s claimant name; and
  `getValidAirlineIds()` (→ `GET /api/internal/airlines`, cached indefinitely per-process since
  `Airline` is fixed seed data with no management UI) for `expense-fields.ts`'s airline-picker field
  validation. Both calls need an `AUTH_SERVICE_URL`/optional `AUTH_SERVICE_INTERNAL_API_KEY` (see
  `.env.example`), same shared-secret pattern as the other two internal services this backend calls.

### Request flow
`src/server.ts` calls `sequelize.authenticate()` then `createApp()` from `src/app.ts`, which wires middleware (helmet, cors, morgan, json body parsing) and mounts `apiRouter` under `/api`. Routing fans out through `src/routes/index.ts` → per-resource router (e.g. `health.routes.ts`) → controller function in `src/controllers/`. Controllers are plain functions `(req, res) => void` — no service/repository layer exists yet beyond the HTTP clients noted above; as more resources are added, follow the same routes → controller → model shape unless the task calls for a new layer. Errors fall through to `notFoundHandler`/`errorHandler` in `src/middleware/error-handler.ts` (registered last in `app.ts`), which return JSON `{ error }` shapes.

### Two separate config sources — keep them in sync manually
DB connection settings are defined **twice** and must be kept consistent by hand:
- `src/config/env.ts` — reads `process.env` via `dotenv`, used by the app at runtime (`server.ts`, `config/database.ts`).
- `src/config/config.js` — plain JS read directly by `sequelize-cli` (per `.sequelizerc`), since the CLI doesn't run through ts-node.

If you add a new env var the app needs, decide whether `config.js` also needs it (only required for values Sequelize CLI itself uses, i.e. DB connection params).

### Models and migrations
Migrations/seeders are plain `.js` (CLI-only, see `.sequelizerc`); models and all other app code are `.ts`. Columns are camelCase end-to-end (`createdAt`/`updatedAt`, not snake_case) — migrations and `Model.init()` column defs must match exactly, and there's no `underscored: true` anywhere. New models: add a migration in `src/migrations/`, define the class in `src/models/`, then re-export it from `src/models/index.ts`. The auth-domain migrations (everything before `20260708190000-create-category-management-tables.js`) are deliberately left in place in `src/migrations/` even though their tables' model files are gone — they're historical record and a rollback safety net for this backend's own database, per the microservices plan's "keep old tables un-dropped for one release cycle" guidance; don't delete them.

### `src/utils/`
What's left here after Phase 3: `category-snapshot.ts`, `category-versioning.ts`, `category-policy-persistence.ts`, `category-policy-validation.ts`, `trip-total.ts`, `duplicate-expense-check.ts`, `expense-fields.ts`, `formula-evaluator.ts`, `invoice-file-storage.ts`, `pdf-pages.ts`, `split-request-mailer.ts`, plus `jwt.ts` (verify-only now — see below) and `utils/constants/` (`category.constant.ts`, `claim.constant.ts`, `trip.constant.ts`). `auth.ts`, `otp.ts`, `password.ts`, `mailer.ts`, `employee-invite-mailer.ts`, `sms.ts`, `employee-bulk-file.ts`, `validation.ts`, and their constants moved to `auth-service` along with everything that used them.

### `jwt.ts` is verify-only — this backend no longer issues any tokens
Login/registration/onboarding all moved to `auth-service`, so `utils/jwt.ts` only exports
`verifyAccessToken`/`AccessTokenPayload` now — no signing functions, no other token types (no more
`password_reset`/`registration`/`onboarding`/`refresh` payloads to verify, since nothing here ever
checks those). `JWT_SECRET` must match `auth-service`'s (and `frontend/proxy.ts`'s) byte-for-byte; a
mismatch means every request here 401s rather than granting access.

### Category Management (013-016) — the largest feature in this codebase, a 4-step wizard plus versioning

`category.routes.ts` (mounted at `/api/categories`) fans out to five controllers, one per wizard step plus supporting concerns: `category.controller.ts` (Step 1 Basic Details, the shared detail/listing/lifecycle endpoints), `category-fields.controller.ts` (Step 2 Expense Form), `category-policies.controller.ts` (Step 3 Claim + Exception Policies), `category-project-policies.controller.ts` (Step 4 Project Policies + activation), and `category-versions.controller.ts` (016's version read endpoints + the edit-session-end trigger).

**No `requireOwner` gate** — any authenticated employee in the organization can create, view, edit, delete, and fully configure any category, including its policies/approval flows, not just categories they created themselves. Every handler reads `req.organizationId` directly (see "Cross-service reads" above), and every controller's request type is plain `AuthenticatedRequest`. `category.controller.ts`'s version-listing endpoint resolves each version's "created by" name via `auth.service.ts`'s `lookupEmployees` now, not a local `Employee` query.

Per `013`'s own framing: **Step 1 is read/written through one shared endpoint every other step also reads from (`GET`/`PATCH /api/categories/:id`); Steps 2-4 each persist through their own independent `PUT` endpoint** (`.../fields`, `.../policies`, `.../project-policies`) — there's no cross-step orchestration or transaction spanning multiple steps' saves.

**`CategoryPolicy` is one flat, `policyType`-discriminated table (`"claim" | "exception" | "project"`), not three separate tables or a nested hierarchy** — confirmed at explicit request after an early draft incorrectly nested Exception Policies under Claim Policies; the three types are fully independent lists. Claim Policies are always required (≥1); Exception Policies are always optional (0 is fine); Project Policies are required (≥1) only when `Category.enableProjectPolicies` is `true`. `category-policy-validation.ts`'s `validatePolicy(policy, options: PolicyValidationOptions)` is the one shared validator behind all three — it varies by `allowedEligibilityTypes` (Claim: department/grade/project; Exception & Project: employee/project respectively, one type each), `maxRuleLevel` (Claim: unlimited; Exception & Project: 1), and `checkRuleDuplicates`, rather than duplicating ~150 lines of eligibility/rule/approval-flow validation three times. `category-policy-persistence.ts` holds the matching shared `persistPolicy`/`buildFieldLookups` write-side helpers, split into their own util (not left inside a controller) specifically so both `category-policies.controller.ts` and `category-project-policies.controller.ts` can import them without one controller importing another.

**Two distinct replace strategies are used deliberately, not inconsistently**: `CategoryField` rows are replaced **by id** (update existing rows in place, create new ones, delete ones no longer submitted) in `saveCategoryFields`, because `CategoryPolicyRule.fieldId`/`comparisonFieldId`/`amountFieldId` and other fields' `conditionalVisibility.dependsOnFieldId` reference a field by id, and `013` requires renames to auto-propagate rather than break those references. `CategoryPolicy` rows (Step 3 and 4) use full **destroy-then-recreate** instead, since nothing outside a policy ever references its id — version snapshots store policy data by value, not by id.

**Field renames auto-update every formula/red-flag reference in the same save request** — `saveCategoryFields` builds a `renameMap` (old name → new name, matched by id between the incoming payload and the existing rows) and runs a string-substitution pass (`applyFieldRenames`) replacing `{{OldName}}` with `{{NewName}}` across every field's `config.formula` and `redFlagValue` before validating. This works only within a single save request, which is the only case `013`'s acceptance criteria actually require.

**Approval flow AND/OR semantics**: `CategoryApprovalStageApprover.logicGroup` groups approvers sharing the same integer within one stage as OR'd (any one of them suffices); different `logicGroup` values within the same stage are AND'd (every group needs its own approval). `CategoryApprovalLevel.isDefaultFlow` is the flag distinguishing the always-present "Default Flow" from the up-to-5 optional numbered Levels (`level: 1-5`, `null` when `isDefaultFlow` is true) — `validatePolicy` enforces exactly one Default Flow per policy.

**Step 4's `PUT .../project-policies` is the confirmed terminal call — there is no separate submit/activate endpoint.** Every successful call flips `Category.status` from `"draft"` to `"active"` (whether `enableProjectPolicies` is `false` with no policies, or `true` with a valid non-empty list), and — per `015`'s Edit flow — the exact same endpoint, called again later against an already-`active` category, just updates Project Policies in place without re-triggering activation (guarded by capturing `wasDraft` before mutating `status`, so version `1.0` creation only fires on the genuine draft→active transition, never on a later edit).

**016's versioning engine (`category-versioning.ts`) is triggered two ways, never per individual step-save**: `createCategoryVersion(category, createdBy)` is called (a) from `saveCategoryProjectPolicies` exactly once, on the draft→active transition, unconditionally creating `1.0` with `modifiedSteps: []` (there's no prior version to diff against); and (b) from `finishCategoryEditSession` (`POST /api/categories/:id/finish-editing`), which the frontend wizard calls once when an edit session for an *already-active* category ends (Cancel, navigate away, or close) — **this endpoint is this story's single most significant architectural resolution**: `016`'s own Open Questions flags that per-step-save versioning is structurally incompatible with the "3+ of 4 steps changed = major" rule, since no individual step's save endpoint ever sees more than its own one step's diff, so a dedicated session-end signal was added rather than inferring versioning from any existing save call. `createCategoryVersion` diffs the freshly-built current snapshot against the latest `CategoryVersion.snapshot` across all 4 step-slices (`basicDetails`/`expenseForm`/`policiesAndApprovals`/`projectPolicies`, see `WIZARD_STEPS`); 0 differing steps creates nothing, 1-2 is a minor bump (`X.Y` → `X.(Y+1)`), 3-4 is a major bump (`X.Y` → `(X+1).0`).

**Version diffing uses a recursive key-sorted `stableStringify`, not plain `JSON.stringify`** — this was a real bug caught during implementation-time verification: Postgres's `jsonb` column type does not preserve object key insertion order, so a snapshot freshly built from live relational data and that same snapshot after a round-trip through the `CategoryVersion.snapshot` JSONB column can hold identical data with differently-ordered object keys, which a naive `JSON.stringify` comparison flags as a spurious diff (observed firsthand: an edit touching only Basic Details was incorrectly computed as a major version until this was fixed). `stableStringify` sorts object keys recursively before serializing but leaves array order untouched, since every array in a `CategorySnapshot` is already meaningfully ordered (by `position`/`level`/`stageNumber` — see `category-snapshot.ts`).

`category-snapshot.ts`'s `buildCategorySnapshot(category)` is the one shared "assemble everything" builder behind `GET /api/categories/:id`, `GET .../versions/latest`, `GET .../versions/:version` (for a still-draft category, which has no real `CategoryVersion` rows), and `createCategoryVersion`'s own snapshot capture — all four read paths return the identical shape by construction, batching queries per policy type via `Promise.all` rather than issuing N+1 queries per policy/level/stage.

### Trip Management (018-019) — no `requireOwner` gate, and a real open question about `Trip.status`

`Trip`/`Country`/`City` (`trip.routes.ts`/`country.routes.ts`/`city.routes.ts`, mounted at `/api/trips`/`/api/countries`/`/api/cities`) are this codebase's first employee-facing resource that's gated by `requireAuth` alone — **no `requireOwner`**, since a trip is every employee's own personal travel record, not a Company-Administrator config screen (`018`'s own explicit posture, a deliberate departure from every other multi-row resource in this codebase). `trip.controller.ts`'s `listTrips`/`createTrip`/`deleteTrip` all scope by `employeeId: req.userId`, not `organizationId` — `organizationId` is still stored on `Trip` (for a future org-wide report), but no query in this epic ever filters by it.

`Country`/`City` are global, seeded-once, no-management-UI catalogs — the same shape `Airline` established (now in `auth-service`), right down to seeding via `bulkInsert` directly inside the creating migration (`20260709120000-create-trip-management-tables.js`) rather than a separate seeder file. **The city seed is a curated "major cities per country" list (~48 countries, ~200 cities), not a full world-cities import** — a genuinely complete dataset is tens of thousands of rows and would realistically come from a public geographic dataset (e.g. GeoNames) rather than being hand-authored; this was an explicit, flagged scope decision (see `018`'s Open Questions), not an oversight. `city.controller.ts`'s `listCities` matches search text against city name **or** country name (so searching "India" surfaces every Indian city) by first resolving matching `Country` rows, then combining `{ countryId: matchingCountryIds } OR { name: iLike }` via `Op.and`/`Op.or` composed as an array of `WhereOptions`.

**`Trip.status` supports four values (`"draft" | "new" | "pending_for_approval" | "approved_for_reimbursement"`) even though this epic's own code only ever writes `"new"`.** The other three exist in the type/column because `019`'s own reference screenshot showed real sample data in those states — but nothing in `018`/`019` builds what would actually produce `"pending_for_approval"`/`"approved_for_reimbursement"` (a future Claims/Approval epic's concern), and **`"draft"` specifically is a known, unresolved gap**: `018`'s Create Trip has no Save-as-Draft action, so nothing in this codebase can currently create a Draft trip, even though the delete endpoint (`deleteTrip`) already enforces "only draft trips can be deleted" defensively, and the column/type support it. Don't build a Save-as-Draft path without re-confirming this is actually wanted first — see `018`'s Open Questions.

`Trip.totalAmount`/`approvedAmount` are `DECIMAL(12,2)` columns — Sequelize returns these as strings, not numbers (avoiding float-precision issues with money), so `trip.model.ts` types them as `string`/`string | null`, not `number`. `approvedAmount` is always `null` until a future story populates it. **`totalAmount` (and `status`) are no longer untouched, though** — `utils/trip-total.ts`'s `recomputeTripFromLinkedClaims(tripId)` (added alongside Claim Management, not part of `018`/`019` themselves) sums every `Claim.totalAmount` linked to a trip via `tripId` (draft and submitted claims both count, matching how `Claim.totalAmount` itself makes no such distinction) and writes the result back to `Trip.totalAmount`; the same call also flips `Trip.status` from `"new"` to `"pending_for_approval"` the first time that sum becomes positive — a one-way transition, never reverted back to `"new"` if a claim is later deleted and the total drops back to 0. `claim.controller.ts` calls it after `saveExpenses` (whenever a trip-linked claim's own total just changed), `deleteClaim` (a trip-linked draft claim being removed), and `splitClaim` (which can leave the original and the new claim linked to two different trips, so both get recomputed independently). Since `trip-select.tsx`'s picker only ever offers `status: "new"` trips, a trip that's already picked up its first expense drops out of that picker — a deliberate consequence of the status transition, not a separate rule. `getTripDetail` also now returns a flattened `expenses` array — every `Expense` across every `Claim` with this `tripId`, one row per expense (category name, claim name, date, amount) rather than grouped by claim — read live off `Claim.tripId` rather than any new join table, so claims linked before this existed show up with no backfill needed.

`listTrips`'s `tripStartDate`/`createdDate` filters are **exact-calendar-day matches**, computed via a `dayRange(value)` helper that builds a `[startOfDay, startOfNextDay)` range and filters with `Op.gte`/`Op.lt` — not a single exact-timestamp match (which would almost never match anything, since `startAt`/`createdAt` carry a time-of-day) and not an open-ended range.

**`Claim` and `Trip` are both `paranoid: true` (soft delete)** — `deletedAt` columns added via `20260714150000`/`20260714150001`, `claim.model.ts`/`trip.model.ts` both set `paranoid: true`. `deleteClaim`/`deleteTrip`'s own `claim.destroy()`/`trip.destroy()` calls now just set `deletedAt` instead of removing the row; every default Sequelize query (`findAll`/`findOne`/`findByPk`/etc.) against either model automatically excludes a soft-deleted row with no call-site changes needed anywhere in this codebase — pass `{ paranoid: false }` to a query to see through it (nothing does today). **This is why `deleteClaim` now explicitly hard-deletes its children first**: the DB's own `ON DELETE CASCADE` from `expenses`/`claim_invoice_files` → `claims` only fires on a real `DELETE`, which paranoid mode never issues — so `deleteClaim` now runs `deleteInvoiceFile()` (physical file) + `ClaimInvoiceFile.destroy` + `Expense.destroy` itself (both still plain hard-delete models) before soft-deleting the `Claim` row, which still cascades to `expense_split_requests` at the DB level exactly as before, since that child's own real `DELETE` still fires. `AiExtractionLog` rows in `ai-service`'s own MySQL database are **not** cleaned up by this cascade anymore — a real, accepted gap now that the table lives in a different database entirely (see "Claim Management" below). `deleteTrip` needed no equivalent change — `claims.tripId → trips.id` is `ON DELETE SET NULL`, not `CASCADE`, and `deleteTrip` only ever runs on a `"draft"`-status trip, which no code path in this app currently links any claim to.

### Claim Management (022-024) — a Category's own field configuration IS the expense form, a real AI/ML integration, and two amendments to already-shipped Category Management

`Claim`/`Expense` (`claim.controller.ts`, mounted at `/api/claims`) and the AI-specific pieces (`claim-ai.controller.ts`, same router) are gated by `requireAuth` alone, no `requireOwner` — same posture as Trip Management, a claim is every employee's own record. `GET /api/categories/claimable` is a narrower sibling of `013`'s own listing endpoint — active + enabled categories only, with each category's `CategoryField[]` inlined (since the claim form needs the actual field configuration to render, not just id/name). It lives on its own tiny `categoryClaimableRouter`, mounted at the same `/categories` prefix but registered *before* `categoryRouter` in `routes/index.ts` (Express tries mounted routers in registration order; the first one whose path matches wins) — the same "narrow router mounted first" trick, not a new pattern.

**A Claim's expense form is not fixed — it's whatever the selected Category's own Step 2 field configuration says it should be.** `expense-fields.ts`'s `validateExpenseFieldValues(categoryFields, fieldValues, isDraftSave)` re-validates submitted values against the same `CategoryField[]` rows `category-fields.controller.ts`'s `validateFieldConfig` already validates at category-*configuration* time — one shared source of truth (the `CategoryField` row), two different moments it's checked. It also resolves `conditionalVisibility` (a field not currently visible is skipped entirely, not required), validates `airline`-type field values against `auth.service.ts`'s cached `getValidAirlineIds()` (see "Cross-service reads" above — this used to be a local `Airline.findAll()` query), and recomputes any `amount`/`number` field with a `config.formula` set via `formula-evaluator.ts` — a small hand-written recursive-descent arithmetic parser (numbers, `+ - * / ( )`, `{{FieldName}}` tokens already substituted), **not `eval()`/`new Function()`**, per this codebase's security rules. The function returns the three denormalized `Expense` columns (`amount`, `expenseDate`, `invoiceNumber` — read off whichever fields carry `useAsClaimAmount`/`useAsExpenseDate`/`useAsInvoiceNumber`) alongside every field's normalized value, and is reused identically by the manual flow's `saveExpenses` and the AI flow's own per-source extraction save.

**Two real amendments to already-shipped Category Management (013), both additive, no backfill needed:**
- `CategoryField.config.useAsInvoiceNumber` (boolean, optional) — mirrors `useAsClaimAmount`/`useAsExpenseDate`'s pattern but, unlike those two, **not** required to be set exactly once per category; zero is fine (a mileage/per-diem category simply has no invoice-number concept, and duplicate detection just never checks it).
- **`city_list` fields now draw from Trip Management's real `Country`/`City` catalog instead of the old hardcoded 10-name `CATEGORY_CITY_LIST`**. `category-fields.controller.ts`'s `validateFieldConfig` now takes a `cityCount` parameter (real `City.count()`) instead of comparing against the old array's `.length`.

**Duplicate Bill Detection (`duplicate-expense-check.ts`'s `findDuplicateExpense`) matches on Expense Date + Invoice/Bill Number + Amount, organization-wide, across every employee — highlighted, never blocking.** It returns `null` immediately if `invoiceNumber` is `null` rather than falling back to a weaker Date+Amount-only match. `expenses`' own composite index `(organizationId, invoiceNumber, expenseDate, amount)` backs this without a full-table scan. The claimant's name in the returned match is now resolved via `auth.service.ts`'s `lookupEmployees` (a single-id lookup) instead of a local `Employee.findByPk`.

**`Expense.categoryId` is nullable** — a manually-created expense always has it set by the time `saveExpenses` persists it, but an AI-created expense can arrive with no confidently-suggested category and sit blank in the Review screen until the employee picks one.

**The AI/ML service is a genuinely separate backend microservice (`../ai-service`), not a module inside this codebase, and it's Python, not TypeScript.** The wire contract (`GET /api/health`, `POST /api/extract`, same request/response JSON shapes) hasn't changed. See that project's own `CLAUDE.md` for its Python-side architecture. `services/ai-extraction.service.ts` here is just a thin HTTP client. **A single invoice source is sent to the model as a native PDF document or image, never rasterized to an image ourselves** — `pdf-pages.ts` (built on `pdf-lib`) only ever needs to read a PDF's page count at upload time and extract pages into a new standalone PDF for Merge.

**`AiExtractionLog` moved to `ai-service`'s own MySQL database in Phase 4 — this service no longer writes it at all.** `runSourceExtraction` in `claim-ai.controller.ts` used to create the log row (`status: "pending"`) before calling the service and update it to `"completed"`/`"failed"` after; it now just passes two correlation keys (`claimInvoiceFileId`, `pageNumber`) to `ai-extraction.service.ts`'s `extractInvoiceData`, and `ai-service` writes its own audit row around its own `/api/extract` call (it owns the table now — every call is still logged regardless of outcome, just by the other side of the HTTP boundary). What used to be a local `AiExtractionLog.findAll` query in two places — `runExtractionForClaim`'s "already logged" dedup check and `getProcessingStatus`'s polling — is now `ai-extraction.service.ts`'s `listExtractionLogs(claimInvoiceFileIds)`, calling `ai-service`'s `GET /api/extraction-logs`. **Processing still runs as an unawaited background task, polled via `GET /:id/process-status`** — that shape is unchanged, only where the status data lives moved.

**Merge folds N per-page expenses into one re-extracted expense; Unmerge re-runs extraction per page from scratch rather than restoring cached values.** `Expense.mergedFromExpenseIds` (JSONB) is repurposed to hold the **original page numbers**, not literal expense ids.

**Uploaded invoice files live on local disk**, not cloud storage — `invoice-file-storage.ts` writes to `backend/uploads/claims/<claimId>/` (gitignored). Files are served back only through an authenticated `GET /:id/invoice-files/:fileId/content` handler, never a public `express.static` path.

**Every existing organization is seeded with 4 sample categories** (`20260710120000-seed-default-categories.js`: Meals, Travel/Flight, Domestic Hotel, Fuel/Local Conveyance) — each exercises a different mechanic: Meals carries an AI-mode red flag field; Travel/Flight uses `list`/`city_list`; Domestic Hotel's Amount is formula-computed from Number of Nights × Rate Per Night; Fuel/Local Conveyance deliberately has no field marked `useAsInvoiceNumber`. **This migration only backfills organizations that already existed at the time it ran** — a brand-new organization created after this shipped starts with zero categories again (a known, accepted gap, since `createRegistration` now lives in `auth-service` and this backend has no hook into it).

### Split Claim (025) — cross-employee expense sharing, not to be confused with 022's own "Split a Claim"

`025` is a genuinely different feature from `022`'s "Split a Claim" (`splitClaim` in `claim.controller.ts`) despite the near-identical name: that one moves an employee's own expenses onto a new claim they still own; `025` shares one expense's *cost* with colleagues in the same organization, who each get their own inbox entry to Accept (raising a real new claim in their own name for their own share) or Reject. To avoid the collision, `022`'s feature was renamed in code comments and its frontend button/dialog labels to **"Move to New Claim"** — no route/API path changed.

**One `ExpenseSplitRequest` = one original `Expense`, not a bundle of several.** This collapses per-expense accept/reject and bulk accept-all/reject-all into a single accept/reject per request (`POST /api/split-requests/:id/accept`, `POST /api/split-requests/:id/reject`).

**Two tables**: `ExpenseSplitRequest` (`expenseId`, `requestedByEmployeeId`, `splitType: "percentage"|"amount"`) and `ExpenseSplitRequestMember` (`splitRequestId`, `employeeId`, `percentage`, `amount`, `isRequester`, `status: "pending"|"accepted"|"rejected"`, `respondedAt`, `resultingExpenseId`) — the requester gets their own `isRequester: true` row too (immediately `"accepted"`, no response needed). Both tables cascade-delete off their parent `Expense`/`ExpenseSplitRequest`.

**`createSplitRequest`** (`POST /api/claims/:id/expenses/:expenseId/split-requests`) requires the claim to still be `"draft"` and the expense to already carry a `categoryId` and a positive amount. Colleagues are validated to belong to the caller's own organization (via `auth.service.ts`'s `lookupEmployees`, filtered client-side by `organizationId` since the internal endpoint itself doesn't scope by org — see "Cross-service reads" above), 1-9 of them, no duplicates, no self-inclusion; percentages must sum to ≤100 (amounts to ≤ the expense's own amount), with the requester's own share computed as whatever remains. Sends one best-effort email per invited colleague (`utils/split-request-mailer.ts`, via `communications-service` — a send failure doesn't block the request itself) linking to `${env.corsOrigin}/claims/split-requests`.

**`acceptSplitRequestExpense`/`rejectSplitRequestExpense`** (named `acceptSplitRequest`/`rejectSplitRequest` in `split-request.controller.ts`) only ever act on the caller's own non-requester member row, 404ing otherwise. Accept requires the same Claim Type fork (`standalone`/`trip_linked`) `022`'s own claim creation uses, and creates a brand-new `Claim` (`creationMethod: "split"`, `hasBeenSaved: true`) plus one new `Expense` copying the original's category/date/invoice-number, with `fieldValues` overriding whichever field carries `useAsClaimAmount` to this member's own share.

**`listSplitRequests`** (`GET /api/split-requests?scope=received|sent`) — `received` (the default) returns the caller's own inbox, with requester names now resolved via `auth.service.ts`'s `lookupEmployees` rather than a local `Employee` query; `sent` returns the requester's own view.
