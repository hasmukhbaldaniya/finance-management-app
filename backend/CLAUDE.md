Implements login/forgot-password (`user-stories/001-authentication.md`), company registration (`user-stories/002-organization-signup.md`), Grade Management (`user-stories/004-grade-management.md`), screen plus its Resend Invitation/Suspend-Activate actions (Story 2 of 3, `user-stories/009-employee-listing.md`), Bulk Invite Employees — CSV/XLSX upload, row-by-row validation, create-or-update import (Story 3 of 3, `user-stories/010-bulk-invite-employees.md`), the invited employee's own Onboarding flow — password, profile, mobile OTP, then landing on the dashboard (Story 4, added after the fact, `user-stories/011-employee-onboarding.md`), Employee Profile — the logged-in employee's own self-service view/edit/change-password screen, a separate epic from Employee Management (`user-stories/012-employee-profile.md`), Category Management — a 4-step Create Category wizard, the My Categories listing with lifecycle actions, Edit/Duplicate, and version history (`user-stories/013-category-creation.md` through `016-category-version-history.md`, see "Category Management" below), Trip Management — Create Trip plus the "My Trips" listing/search/filter/delete (`user-stories/018-trip-creation.md`/`019-trip-listing.md`, see "Trip Management" below), and Claim Management — Manual Add Claim, the AI-Powered (Automated Extraction) flow with its own AI/ML service, and the "My Claim" listing (`user-stories/022-claim-creation-manual.md` through `024-my-claim-listing.md`, see "Claim Management" below) — no service/repository layer yet (except the AI/ML extraction service, its own genuinely separate concern), controllers talk to models directly. `003-header-navigation.md`'s switch-active-organization feature is documented there but no longer implemented — see "The User → Employee merge" below for why.

Models: `Employee` (the login/session entity — see "The User → Employee merge" below for how it absorbed `User`; also the entity Employee Invitation, `008`, is built around), `EmployeeCompanyAccess` (per-employee Role/Department/Grade assignment — also the successor to the removed `OrganizationMember`), `Organization`, `Otp` (a single generalized table for all three OTP purposes — password reset, email verification, mobile verification — distinguished by a `purpose` column; see "OTPs are generalized" below), `Grade`/`Department`/`Role` (organization-scoped lookup entities, see below — `Role` additionally carries `isDefault` and a `privileges` string array, see "Roles & Privileges" below), `AssociatedOrganization` (an org-to-org network row, not org-scoped-lookup like the others — see "Associated Organizations" below), `Project`/`EmployeeProject`/`Airline`/`EmployeeFfNumber`/`ApprovalLevel`/`EmployeeInvite` (the remaining tables backing Employee Invitation — see "Employee Invitation" below), `BulkUpload`/`BulkUploadError` (audit trail for `010`'s upload validation attempts — see "Bulk Invite Employees" below), `Category`/`CategoryZiptrripMapping`/`CategoryField`/`CategoryPolicy`/`CategoryPolicyEligibility`/`CategoryPolicyRule`/`CategoryApprovalLevel`/`CategoryApprovalStage`/`CategoryApprovalStageApprover`/`CategoryVersion` (Category Management's 10 tables — see "Category Management" below), `Country`/`City`/`Trip` (Trip Management — see "Trip Management" below), `Claim`/`Expense`/`ClaimInvoiceFile`/`AiExtractionLog` (Claim Management — see "Claim Management" below). Employee Onboarding (`011`) adds no new tables — it writes to the same `Employee` columns the User → Employee merge already added, and reuses `Otp` under a new `purpose` value.

Email (OTP delivery) goes through `nodemailer` in `src/utils/mailer.ts`, configured via `SMTP_HOST`/`SMTP_PORT`/`SMTP_USER`/`SMTP_PASSWORD`/`SMTP_FROM` in `.env` (see `.env.example`; defaults point at Gmail's SMTP host/port, but `SMTP_USER`/`SMTP_PASSWORD` have no fallback and must be set locally — Gmail requires a 16-char App Password, not the account password). `sendOtpEmail`'s third argument picks the subject line per purpose (`"password_reset"` | `"email_verification"`). Mobile OTP delivery (`src/utils/sms.ts`) is a `console.log` dev stub — no SMS provider is configured, same situation `mailer.ts` was in before Gmail SMTP was wired up; swap it for a real provider (Twilio/MSG91/etc.) before production. No queue/retry — send failures currently throw straight out of the controller.

## Architecture

Structure: `src/{config,models,controllers,routes,middleware,migrations,seeders}`.

### The User → Employee merge — `Employee` is the login entity, `User`/`OrganizationMember` no longer exist
This codebase used to have a separate `User` (login/session/OTP-verification/multi-org-membership) and `Employee` (008's richer per-org person record — DOB, gender, approval chains, company access). At explicit request, they were merged: `Employee` absorbed everything `User` did, and `User`/`OrganizationMember` were dropped entirely via a migration sequence (`20260707140000` through `20260707170000`) that added auth fields to `employees`, relaxed several columns to nullable, data-migrated every `users` row into a matching/new `employees` row (repointing `createdBy`/`updatedBy`/`sentBy` FKs from old user ids to the new employee ids along the way), then dropped `organization_members` and `users` and replaced `employees`' org-scoped email/contact-number unique indexes with global ones (an `Employee` is now a login identity, and login has no organization context to scope uniqueness by).

**What this means concretely:**
- `Employee` gained `passwordHash`, `emailVerifiedAt`, `mobileVerifiedAt` (all nullable — a still-pending invitee has none of these yet), and `isOwner` (boolean, replacing `OrganizationMember.role === "owner"`).
- `title`/`gender`/`countryCode`/`contactNumber`/`createdBy`/`updatedBy` became nullable, since self-registration (`registration.controller.ts`) collects far less than the admin-invite wizard does (008) — no title/gender/contact number upfront, and no inviting admin to record. The invite wizard's own application-level validation still requires those fields for admin-created rows; only the DB constraint relaxed.
- `Employee.userId` was removed outright — there's no separate row to link to anymore, the Employee row *is* the account.
- `invitationStatus` (`"pending"|"registered"`) now doubles as what `User.registrationCompletedAt` used to gate — `requireRegistrationUser`'s single-use-token guard checks `invitationStatus === "registered"` instead of a timestamp column, rather than keeping two fields that mean the same thing.
- `EmployeeCompanyAccess.departmentId`/`gradeId` became nullable (`roleId` stays required) so a freshly self-registered org creator can get a `Company Admin` `EmployeeCompanyAccess` row immediately (needed for the approver picker, see below) without a Department/Grade existing yet.
- **Multi-organization membership is gone.** `OrganizationMember` (the many-to-many join table, `User`↔`Organization`) is dropped along with `User` — `Employee.organizationId` is fixed at creation, so there's no more "active organization" to switch. This deleted `PATCH /api/users/me/active-organization`, `GET /api/organizations/mine`, `user.controller.ts`/`user.routes.ts` entirely, and the frontend's switch-organization screen (`003`'s feature). `007`'s **different** "Associated Organizations (Network)" feature — an org-to-org listing, not personal multi-membership — is unaffected.
- `grade.controller.ts`/`department.controller.ts`/`role.controller.ts`'s `list*Members` endpoints and `membersCount` aggregates moved from querying `OrganizationMember`+`User` to `EmployeeCompanyAccess`+`Employee` — same shape, different source table.
- `utils/auth.ts`'s helpers were rewritten against `Employee`: `toPublicUser` → `toPublicEmployee` (still returns `{id, name, email, phone}` — `name` is derived as `` `${firstName} ${lastName}`.trim() ``, `phone` as `contactNumber`, so the frontend's response contract is unchanged), `getCurrentOrganization` now just does `Organization.findByPk(employee.organizationId)` (no more membership-fallback lookup), `getActiveOrganizationId(userId)` now resolves via `Employee.findByPk(userId)?.organizationId` (signature unchanged — every call site across `grade`/`department`/`role`/`project`/`employee` controllers needed zero changes), `isOrganizationOwner` now checks `employee.isOwner` directly.
- `middleware/require-owner.ts` simplified to `Employee.findByPk(req.userId)` + check `.isOwner` — no more separate `OrganizationMember` lookup.
- `employee.controller.ts`'s `ensureSelfEmployee` — a function that used to fabricate a placeholder `Employee` row (fake `title:"Mr"`, `gender:"Other"`, synthetic contact number) for a logged-in admin who had no real `Employee` row — was deleted outright. Every logged-in principal is a real `Employee` row now, so there's nothing left to fabricate; `listEmployeesForPicker` just always includes `req.userId` in its result set.
- **Real pre-existing data note**: the data migration found one genuine conflict during this merge — an `Employee` row in one organization with `invitationStatus: "pending"` (a never-accepted invite) shared an email with a different, real `User` account being migrated in from another organization. Since `Employee.email` is now globally unique, that stale, never-accepted invite was deleted as part of the migration (it had no password, no completed onboarding — it couldn't coexist with the real account under the new uniqueness rule). Any similarly-stale pending invites sharing an email with a real account would hit the same resolution if this migration sequence were ever re-run against different data.

### Request flow
`src/server.ts` calls `sequelize.authenticate()` then `createApp()` from `src/app.ts`, which wires middleware (helmet, cors, morgan, json body parsing) and mounts `apiRouter` under `/api`. Routing fans out through `src/routes/index.ts` → per-resource router (e.g. `health.routes.ts`) → controller function in `src/controllers/`. Controllers are plain functions `(req, res) => void` — no service/repository layer exists yet; as more resources are added, follow the same routes → controller → model shape unless the task calls for a new layer. Errors fall through to `notFoundHandler`/`errorHandler` in `src/middleware/error-handler.ts` (registered last in `app.ts`), which return JSON `{ error }` shapes.

### Two separate config sources — keep them in sync manually
DB connection settings are defined **twice** and must be kept consistent by hand:
- `src/config/env.ts` — reads `process.env` via `dotenv`, used by the app at runtime (`server.ts`, `config/database.ts`).
- `src/config/config.js` — plain JS read directly by `sequelize-cli` (per `.sequelizerc`), since the CLI doesn't run through ts-node.

If you add a new env var the app needs, decide whether `config.js` also needs it (only required for values Sequelize CLI itself uses, i.e. DB connection params).

### Models and migrations
Migrations/seeders are plain `.js` (CLI-only, see `.sequelizerc`); models and all other app code are `.ts`. Columns are camelCase end-to-end (`createdAt`/`updatedAt`, not snake_case) — migrations and `Model.init()` column defs must match exactly, and there's no `underscored: true` anywhere. New models: add a migration in `src/migrations/`, define the class in `src/models/`, then re-export it from `src/models/index.ts` (see `employee.model.ts` for the `InferAttributes`/`InferCreationAttributes`/`CreationOptional` pattern).

### `src/utils/`
Flat files (`jwt.ts`, `otp.ts`, `password.ts`, `mailer.ts`, `sms.ts`, `validation.ts`, `auth.ts`) plus one subfolder, `src/utils/constants/`, for pure constant values — currently `regex.constant.ts` (email/phone/OTP/password/GST/name patterns), imported by `validation.ts`'s helper functions rather than inlined there. This mirrors the frontend's `src/utils/constants/` + `src/utils/helpers/` split (see `frontend/CLAUDE.md`) at the scale this backend actually needs — a `helpers/` subfolder isn't split out yet since `validation.ts` is still the only helper module; move it into `src/utils/helpers/validation.helper.ts` if/when a second one shows up. `auth.ts` holds `accessTokenCookieOptions()` and `toPublicEmployee()` — shared between `auth.controller.ts` and `registration.controller.ts` so both issue identical session cookies and the identical public user shape.

### OTPs are generalized, not per-feature
There's one `Otp` model/table for every OTP purpose (`purpose: "password_reset" | "email_verification" | "mobile_verification"`, plus an `identifier` column holding whichever email/phone the OTP was sent to) rather than a dedicated table per feature. When adding a new OTP-gated flow, add a new `purpose` value and filter by it — don't create another table. `Otp` has never had any FK/association to the login entity (identifier is matched by plain string value) — it needed zero schema changes for the User → Employee merge above, only the surrounding controller code's `User.findOne`/`findByPk` calls became `Employee` ones.

### Registration token vs. reset token vs. access token
Three distinct JWT types exist in `src/utils/jwt.ts`, each with a `type` discriminant checked on verify: `access` (the real session, in the httpOnly cookie), `password_reset` (short-lived, issued after forgot-password OTP verification), `registration` (short-lived, issued after registration email-OTP verification — required by every `/api/auth/registrations/*` endpoint from that point on). `registrationToken` is single-use: `completeRegistration` sets `Employee.invitationStatus = "registered"`, and every registration endpoint's guard (`requireRegistrationUser` in `registration.controller.ts`) rejects the token once that's set, even though the JWT itself would otherwise still be valid until its ~15-minute expiry.

### `refreshToken` is issued but not yet redeemable
`POST /api/auth/registrations/complete` returns a `refreshToken` in its response body (per `user-stories/002-organization-signup.md`'s documented contract), but there's no `/api/auth/refresh` endpoint to redeem it yet, and no rotation/revocation policy — building that is separate, not-yet-requested infrastructure. Don't assume a refresh flow exists elsewhere just because a token is being handed out here.

### Org-scoped CRUD resources (Grade, Department, Role, and the siblings still to come)
`Grade` (`grade.*`, mounted at `/grades`), `Department` (`department.*`, mounted at `/departments`), and `Role` (`role.*`, mounted at `/roles`) are the first three of what `user-stories/008`–`010` describe as several near-identical org-scoped CRUD resources. `Employee` ended up needing its own shape rather than a straight copy of these three, since it's a genuinely different kind of entity — see "Employee Invitation" below. The shared pattern for these three: every handler starts by resolving the caller's organization via `getActiveOrganizationId(req.userId)` in `utils/auth.ts` (a small helper added specifically so each new resource's controller doesn't re-fetch the login entity and re-read its organization itself) and 401s if that's null; every row lookup filters by `{ id, organizationId }` together so a request for another organization's row 404s instead of leaking a 403 (see `user-stories/004-grade-management.md`'s Edge Cases for why). Name-uniqueness is checked case-insensitively via `Op.iLike` in application code (a plain unique index would be case-sensitive), matching the same pattern GST/email uniqueness already uses elsewhere. `membersCount` (and the Members-column sort) is computed by aggregating `EmployeeCompanyAccess` rather than stored on `Grade`/`Department`/`Role` themselves, and — since that aggregate (and, for `Role`, the `roleType` sort derived from `isDefault`) can't be ordered inside the same paginated query as the base table — sorting/pagination happens in memory after fetching the full (organization-scoped, search-filtered) result set; this is a deliberate trade-off documented in `004`'s Open Questions, correct at today's expected scale (an organization's grade/department/role list) but revisit if that assumption stops holding. `Grade`/`Department`/`Role` are deliberately separate tables/columns rather than one generalized "lookup" table with a type discriminant — see `005`'s Open Questions for why.

### Roles & Privileges — default roles and the privilege catalog
`Role` is the first org-scoped resource with rows an org didn't create and can't fully control: every organization gets a seeded `Company Admin` (all privileges) and `Members` (a fixed subset) row, both `isDefault: true` — see `utils/constants/role.constant.ts` for the privilege key list (kept byte-identical to the frontend's copy in `src/utils/constants/privilege.constant.ts`) and `20260702110000-create-roles.js` for the seeding/backfill migration. `role.controller.ts`'s `updateRole`/`updateRoleStatus`/`deleteRole` all 403 immediately if `role.isDefault` — enforced server-side regardless of what the frontend's disabled UI already prevents (see `006`'s Edge Cases). New registrations get their two default roles from `registration.controller.ts`'s `createRegistration` (mirroring how it already creates the `Organization`), not from the migration — the migration only backfills organizations that predate this feature. `listRoles` also pins Company Admin/Members ahead of every custom role regardless of `sortBy`/`sortDir` — computed by sorting the default and custom groups separately, then concatenating defaults-first *before* the pagination slice, so they always land on page 1 and never reappear later. Added after the initial implementation, at explicit request — not in the original story spec.

### Associated Organizations — the first role-gated screen, and a different kind of "organization-scoped" resource
`AssociatedOrganization` is **not** an org-scoped lookup like `Grade`/`Department`/`Role` — it's an org-to-org network row (`ownerOrganizationId` is the org that owns the view; `organizationId` is nullable and only populated once an invited contact actually registers). `requireOwner` (`middleware/require-owner.ts`) is the first permission check in this codebase beyond plain authentication: it checks the caller's `Employee.isOwner` flag (shared with `GET /me`'s `isOwner` flag so the frontend can hide the nav link without a separate request) and 403s otherwise, attaching `req.organizationId` so `associated-organization.controller.ts` doesn't re-resolve it. **`Registrations` ("Self-Registered"/"Invited"/"Registered") is computed, not stored** — derived from `registrationType` + whether `organizationId` is set — same "derive, don't duplicate" reasoning as `Role`'s `roleType`/`membersCount`. There's no create/invite endpoint in this story (see `007`'s Out of Scope) — the table starts empty for every organization and stays that way until a future story adds the invite flow; `listAssociatedOrganizations` fetches the full organization-scoped set and filters/sorts/paginates in memory, the same trade-off as the other CRUD resources' derived columns. `listAssociatedOrganizations` also accepts four case-insensitive substring query params (`organizationName`/`contactName`/`contactEmail`/`contactPhone`) applied in the same in-memory pass as the `registrations`/`status` filters — added after a frontend redesign against approved reference screenshots introduced per-column text search (see `007`'s Open Questions); `registrations`/`status` still parse as comma-separated multi-value for forward-compatibility even though the current UI only ever sends 0-or-1 value each.

### Employee Invitation — the module→approval fan-out (and its later removal)
`Employee` (`employee.controller.ts`, mounted at `/employees`) is deliberately **not** a copy of the `Grade`/`Department`/`Role` CRUD shape above — it's a genuinely richer entity (see "The User → Employee merge" above for what it now also covers). The wizard's 4 steps map to 4 handlers, each idempotent-on-resubmit via replace-not-append writes: `createEmployee` (Step 1, org-scoped uniqueness on employeeCode plus **global** uniqueness on email/contact — see the merge note above — same `Op.iLike`-free exact-match style as the unique indexes since these are checked-then-inserted, not case-insensitive lookups like `Grade`/`Department`/`Role` names), `updateEmployeeCompanyAccess` (Step 2, `findOrCreate`-then-update on `EmployeeCompanyAccess` since it's 1:1 per employee, plus a destroy-then-recreate replace of `EmployeeProject` rows — every submitted `projectId` is validated to belong to the submitted `departmentId` first), `addEmployeeFfNumbers` (Step 3, same destroy-then-recreate replace on `EmployeeFfNumber`), and `saveEmployeeApprovals` (Step 4a, same replace pattern on `ApprovalLevel`). `sendEmployeeInvite` (Step 4b) is the terminal action: 409s if `invitationStatus !== "pending"` (already registered), 429s past a daily cap of 5 sends (counted via `EmployeeInvite.count` within a day-boundary window — the same `otpResendCooldownSeconds`-style rate-limit shape already used for OTP resends, just applied to a log table instead of a single-row cooldown), then calls `sendEmployeeInviteEmail` (`utils/employee-invite-mailer.ts` — a real `nodemailer` send as of `011`, see "Employee Onboarding" below for why it's no longer a stub) and logs an `EmployeeInvite` row.

**Module Access was removed after initial implementation, at explicit request** — Step 4's UI no longer has a Trips/Claims/Approvals/Finance/Reports checkbox group, and `saveEmployeeApprovals` no longer reads a `moduleAccess` field from the request body at all. `ApprovalLevel` originally existed to support a chain *per module* (`employeeId, module, level`) with the single submitted chain fanned out across every checked module; since there's no longer a module selection to fan out across, every approval row is now saved under the fixed sentinel `GENERAL_APPROVAL_MODULE = "general"` (`utils/constants/employee.constant.ts`) — one row per approver level, not one per (module × level). `ApprovalLevel.module` still exists as a schema column (dropping it would need a migration, not done here) but is no longer meaningful data, just a constant. See `008`'s Open Questions for the history — Module Access was added, then removed, within this same story's implementation.

**`GET /api/employees/approvers` (`listEmployeesForPicker`) is the approver picker, moved off the main path once `009` needed it**: minimal, unpaginated, privilege-scoped — it returns only active employees whose Role carries the `claim_trip_approvals` privilege (`APPROVER_PRIVILEGE_KEY` in `utils/constants/employee.constant.ts`) — Company Admin has this by default, Members doesn't, but any custom role granting it also qualifies — via a `Role.privileges @> ['claim_trip_approvals']` (`Op.contains`) query joined through `EmployeeCompanyAccess.roleId`. The logged-in caller is always included regardless of that filter — simply `employeeIds.add(req.userId)`, since (per the merge above) every logged-in caller already is a real `Employee` row; there's no more shadow-record fabrication needed here. `GET /api/employees` (no trailing path) is `009`'s full listing instead — see below.

`Project` (`project.controller.ts`, mounted at `/projects`) and `Airline` (`airline.controller.ts`, mounted at `/airlines`) are minor supporting resources added during `008`'s implementation, not originally itemized in that doc's API Design table beyond `POST /api/projects`: `GET /api/projects?departmentId=` and `GET /api/airlines` exist purely to populate Step 2's Projects picker and Step 3's Airline picker. `Airline` is a fixed, seeded, global (non-org-scoped) catalog — no management UI, mirroring `006`'s privilege catalog.

### Employee Listing (009) — the main `GET /api/employees`, self-exclusion, and the last-Company-Admin rule
`listEmployees` (`employee.controller.ts`) is the searchable/filterable/sortable/paginated screen `009` specs, mounted at plain `GET /employees` — a materially different shape from the approver picker above, which is why that one moved to `/employees/approvers` instead of staying on the shared path. **It always excludes the caller's own record** (`id: { [Op.ne]: req.userId }`) — an explicit, later addition beyond `009`'s own doc, since an admin manages their own account elsewhere, not through this table. Role/Department/**Grade** names all come from a join through `EmployeeCompanyAccess` (same as `Grade`/`Department`/`Role`'s own `membersCount`), so sorting and pagination happen in memory after fetching the full filtered set, not inside the paginated query itself — the same documented trade-off as those three controllers. `009`'s own doc only ever specced Role/Department as listing columns; Grade was added to the response shape (and to `SORTABLE_FIELDS`) after the fact, at explicit request, since `updateEmployeeCompanyAccess` already requires and stores `gradeId` for both Invite and Edit (Step 2, see "Employee Invitation" below) — it just wasn't being surfaced back on the listing. The `status` query param is a combined view over two independent columns (`Employee.status` and `Employee.invitationStatus`) per `009`'s own spec: `pending` means `invitationStatus: "pending"` regardless of `status`; `active`/`suspended` mean `invitationStatus: "registered"` and that `status` value.

**Filtering is per-column (`name`/`email`/`contactNumber`), not `009`'s original single combined `search` param** — redesigned after initial implementation, at explicit request, to match `007`'s Associated Organizations filter pattern instead (see that section above): `name` OR-matches `firstName`/`lastName`/`employeeCode` (Employee ID has no dedicated visible column, so it's folded in here rather than dropped), `email` and `contactNumber` each match only their own column, all case-insensitive substring (`Op.iLike`) and combined with `Op.and` alongside the `status` filter and the self-exclusion clause. This is a real divergence from `009`'s documented API Design table (which still shows one `search` param) — not a mistake, a later UI-consistency decision; the doc's Open Questions record it.

`POST /api/employees/:id/resend` is mounted as a second route pointing at the same `sendEmployeeInvite` handler `008` already built — no duplicated logic, since that handler already treats first-send and resend identically (see `008`'s Open Questions for why).

`PATCH /api/employees/:id/status` (`updateEmployeeStatus`) enforces `009`'s two Suspend business rules server-side: self-suspend is rejected outright (`employeeId === req.userId`, checked before even validating the target `status` value further), and suspending an employee is rejected if they hold the organization's `Company Admin` `EmployeeCompanyAccess.roleId` and no *other* employee with that same role is currently `active`/`registered`. This rule is checked purely via `EmployeeCompanyAccess`, not `Employee.isOwner` — a real, known gap: employees migrated in by the User → Employee merge (everyone who predates `008`'s invite wizard) have no `EmployeeCompanyAccess` row at all, so they don't count as "Company Admin" for this rule even though they're `isOwner: true`. This is the same already-documented `OrganizationMember`/`EmployeeCompanyAccess` reconciliation gap from `008`'s Open Questions, not a new one introduced here — flagging it again since this is the first place it has a concrete, testable behavioral consequence.

`GET /api/employees/:id` (`getEmployeeDetail`) and `PATCH /api/employees/:id` (`updateEmployeeBasicInfo`) power the Listing's **Edit** action — a new endpoint pair not in `009`'s original API Design table, added because that doc explicitly defers Edit's implementation-level spec ("reuses 008's field-level rules verbatim... left for implementation time"). `getEmployeeDetail` returns the same shapes `updateEmployeeCompanyAccess`/`addEmployeeFfNumbers`/`saveEmployeeApprovals` already accept, so the frontend's Edit page can round-trip the response straight back into those write calls. `updateEmployeeBasicInfo` mirrors `createEmployee`'s exact validation, with two differences: it excludes the employee's own id from the email/contact-number/employeeCode uniqueness checks, and — like `createEmployee`, fixed in the same pass — checks email/contact-number uniqueness **globally**, not scoped to `organizationId` (that scoping was stale after the merge made those columns globally unique; `createEmployee`'s pre-check had been left un-updated until this fix).

### Bulk Invite Employees (010) — CSV/XLSX upload, validate-then-confirm, and why parsed rows live in memory, not a table
`employee-bulk-invite.controller.ts` (mounted at `/api/employees/bulk`, its own sibling router alongside `employeeRouter` rather than nested inside it — every route here is gated by `requireOwner` on top of `requireAuth`, which nothing else under `/employees` is) implements `010`'s upload → validate → confirm flow: `GET /template` downloads a generated `.xlsx` sample (`utils/employee-bulk-file.ts`'s `buildSampleTemplateBuffer`), `POST /upload` (multipart, handled by `multer` with a 10 MB/`.csv`+`.xlsx`-only `fileFilter`) parses the file, validates every row independently, and returns a summary; `POST /import` persists whichever rows validated; `GET /:uploadId/errors` downloads the per-row failures as a generated `.xlsx`.

**Parsed-but-unconfirmed rows are held in an in-process `Map<uploadId, {organizationId, rows, expiresAt}>`, not a table** — `010`'s own Data Model explicitly says the uploaded file's rows stay unpersisted until `/import` is called, and there's no queue/cache layer anywhere else in this codebase to reach for instead. This is a real, accepted limitation at this app's current single-instance scale: it doesn't survive a server restart, and wouldn't work if this backend ever ran as more than one instance — a genuine production version would need a real store (Redis, or a staging table) instead. `BulkUpload`/`BulkUploadError` (new tables, INTEGER autoincrement ids like every other table in this codebase — `010`'s doc specified UUIDs, changed here for consistency, see that doc's Open Questions) are a separate concern: audit rows created at `/upload` time (`status: "validated"`, flipped to `"imported"` once `/import` runs) that record what happened, independent of whether the parsed rows are still sitting in memory or have already expired.

**Row validation (`validateRows`, the bulk of the controller) is a 3-pass function**, not because the codebase has a validation-layer convention but because the "does this row's email/contact number collide with someone else" check genuinely needs to run in three stages to stay efficient: pass 1 walks every row using data pre-fetched *once* (every employee, active Role/Department/Grade in this organization) rather than issuing per-row queries; rows that pass produce a `create`-or-`update` decision via `010`'s own matching priority (Email → Country Code + Contact Number → Employee ID, scoped to *this* organization); pass 2 issues a single batched query checking whether any surviving "create" candidate's email/contact number is already taken by a *different* organization's employee — a conflict `010`'s own "matching priority" section predates, since `Employee.email`/`(countryCode, contactNumber)` are globally unique now (see the User → Employee merge above), not organization-scoped as that doc assumed; pass 3 finalizes, rejecting any row pass 2 flagged. **`Company` (a per-row required column) is resolved as "must case-insensitively equal the caller's own organization's name," not a cross-organization lookup** — every other endpoint in this codebase scopes writes to the caller's own active organization with no way to target another one, and `010`'s literal spec (a `Company` column implying the row could name a *different* org) doesn't fit that model; this is a documented resolution, not an oversight (see `010`'s Open Questions).

A handful of row-level messages exist beyond `010`'s own Error/Toast Messages table (invalid Gender, duplicate Employee ID within the file, the two cross-organization conflict messages above) — same category of implementation-time addition as `009`'s per-column filters, needed for correctness/consistency even though the source doc didn't spell them out. `utils/employee-bulk-file.ts` also holds `sanitizeCell()`, applied to every cell read from an uploaded file and every user-supplied value written into a generated error report, per `010`'s Security section — a value beginning with `=`/`+`/`-`/`@` is prefixed with a single quote so it can't execute as a formula if the file is later opened in Excel/Sheets. That file uses `exceljs` for `.xlsx` (both reading uploads and generating the template/error report) and `csv-parse` for `.csv` uploads — **not** the `xlsx` (SheetJS) npm package, which has a live, unpatched high-severity prototype-pollution advisory that specifically matters here since this code parses untrusted user-uploaded files; `exceljs`'s own bundled type declarations have an unrelated defect (`declare interface Buffer extends ArrayBuffer {}`, globally corrupting the ambient `Buffer` type for this whole compilation) worked around with a single commented `@ts-expect-error` at its one `xlsx.load()` call site, not a wider type-safety compromise.

### Employee Onboarding (011) — the invited employee's own accept-invite flow, and why it's a sibling router, not a nested one
`011` picks up where `008`'s invite email left off: `sendEmployeeInvite` now mints a short-lived `onboarding` JWT (`signOnboardingToken(employeeId, email)` in `src/utils/jwt.ts`, `ONBOARDING_TOKEN_EXPIRES_IN` env var, default `10m`) and builds a real `http://localhost:3000/onboarding?token=...` link, sent via `utils/employee-invite-mailer.ts` — which graduated from a `console.log` stub to a real `nodemailer` send in this story, for the same reason `utils/sms.ts` *hasn't*: there's finally a real URL worth emailing. `employee-onboarding.controller.ts` (mounted at `/api/employee-onboarding`, **not** nested under `employeeRouter`/`/employees` — that router has a blanket `requireAuth`, and none of these requests carry a session yet, mirroring why `/auth/registrations` stays public as a sub-router of the also-gate-less `authRouter`) implements the 4-step flow: verify-token (also stamps `emailVerifiedAt` the first time a link is opened — there's no separate email-OTP step here, the link itself is the proof of email ownership), password, profile (title/firstName/lastName, pre-filled but editable), mobile (sets `countryCode`/`contactNumber`, globally unique like `createEmployee`'s own check, then sends an OTP), mobile-otp (resend) and mobile-otp/verify (both reusing `Otp` under `purpose: "mobile_verification"`, identical shape to registration's own mobile step), and complete.

**One flat 10-minute token covers the entire 4-step flow** — there's no separate short-lived step-token the way registration's `registrationToken` is reissued after email-OTP verification. Every endpoint re-verifies the same token independently via a private `requireOnboardingEmployee` guard (same "returns `{payload, employee}` or writes the error response itself" shape as `registration.controller.ts`'s `requireRegistrationUser`), and rejects it outright once `employee.invitationStatus === "registered"` — making the token effectively single-use for the flow as a whole, even though the JWT itself would otherwise still be valid until expiry. This means a slow invitee who takes longer than 10 minutes across all 4 steps has to ask for a resend (`009`'s existing Resend Invitation button, unchanged — it already re-mints a fresh token+link each time); see `011`'s Open Questions for why a flat window was chosen over refreshing it per step.

**`completeOnboarding` is the same terminal action regardless of which of the 3 exit points got there** — Step 3's Skip, Step 4's Skip, and Step 4's successful OTP verify all call it, and it does exactly what `registration.controller.ts`'s `completeRegistration` does for self-registration: flips `invitationStatus` to `"registered"`, issues a real access/refresh token pair, sets the httpOnly session cookie, and returns `{user, organization, accessToken, refreshToken}` — the same shape `009`'s listing already expects, so **the Employee Listing screen needed zero changes** for this story: a completed-onboarding employee already shows up there as `invitationStatus: "registered"`, `status: "active"`, with Resend hidden and Suspend/Activate available, purely because `009`'s existing logic keys off those two columns.

### Employee Profile (012) — self-service, not admin management, and why the mobile-change flow doesn't touch the schema
`employee-profile.controller.ts` (`GET`/`PATCH /api/employees/me`, `PUT /api/employees/me/mobile`, `POST /api/employees/me/mobile-otp`, `POST /api/employees/me/mobile-otp/verify`, all mounted on `employeeRouter` — registered *before* the `/:id` routes, same reason `/approvers` is, otherwise Express would match `/me` against `/:id` with `id="me"` first) is deliberately a separate epic from Employee Management (`008`–`011`): those stories are an admin managing *other* employees; this is every employee (including admins) managing their own account, the "elsewhere" `009`'s listing explicitly points to by excluding the caller's own row. `PATCH /api/auth/me/password` (`changePassword` in `auth.controller.ts`, not the employee controller — it's a credential change, living alongside `login`/`resetPassword`) is the third piece, distinguished from `resetPassword` by verifying `currentPassword` instead of an OTP-issued reset token, since this is a still-logged-in employee who remembers their password, not someone locked out.

**`PATCH /api/employees/me`'s body is deliberately narrow** — it reads `title`/`firstName`/`lastName`/`dob`/`gender`/`employeeCode` only; `email`, `organizationId`, `roleId`, `departmentId`, `gradeId` aren't read at all, so there's no reliance on the frontend simply not sending them. Employee ID is genuinely self-editable here (with the same org-scoped uniqueness check, excluding the caller's own current value, as `createEmployee`/`updateEmployeeBasicInfo`) — Organization/Role/Department/Grade are not, since those carry real privileges `009`'s Edit action still gates behind admin access.

**Changing your own Contact Number doesn't touch `Employee.countryCode`/`contactNumber` until the OTP verifies** — a deliberate departure from `011`'s onboarding mobile step, which saves the pending number immediately since it was never previously in use for login. Here the employee already has a working, verified number, so provisionally overwriting it before verification would risk clobbering a good number on a failed or abandoned change. Since `Otp` has no FK to `Employee` by design (see "OTPs are generalized" above), the pending `countryCode`/`contactNumber` pair is encoded directly into `Otp.identifier` as `` `${employeeId}:${countryCode}:${contactNumber}` `` rather than adding new nullable columns to `Employee` for one in-flight value — `setMyMobile`/`resendMyMobileOtp`/`verifyMyMobileOtp` all look up the caller's pending change via `identifier LIKE '${employeeId}:%'` and parse the pair back out of whichever row they find. `verifyMyMobileOtp` re-checks the number's uniqueness at verification time too, not just when the change was started, since another employee could have claimed it in the meantime.

### Category Management (013-016) — the largest feature in this codebase, a 4-step wizard plus versioning

`category.routes.ts` (mounted at `/api/categories`) fans out to five controllers, one per wizard step plus supporting concerns: `category.controller.ts` (Step 1 Basic Details, the shared detail/listing/lifecycle endpoints), `category-fields.controller.ts` (Step 2 Expense Form), `category-policies.controller.ts` (Step 3 Claim + Exception Policies), `category-project-policies.controller.ts` (Step 4 Project Policies + activation), and `category-versions.controller.ts` (016's version read endpoints + the edit-session-end trigger).

**No `requireOwner` gate, at explicit request — a departure from this router's own original posture.** Category Management originally required `requireOwner` on every route (category configuration was treated as at least as sensitive as Associated Organizations/Bulk Invite); this was later opened up so any authenticated employee in the organization can create, view, edit, delete, and fully configure any category, including its policies/approval flows — not just categories they created themselves. Every handler that used to read `req.organizationId` (set by `requireOwner`) now resolves it via `getActiveOrganizationId(req.userId)` instead (the same helper `Trip`/`Claim`/`Grade`/`Department`/`Role` controllers already use), and every controller's request type changed from `OwnerRequest` to plain `AuthenticatedRequest`. This is also what made the `categoryClaimableRouter` workaround (see Claim Management below) no longer strictly necessary for permission reasons — it's kept anyway since it still serves a real, independent purpose (server-side filtering to active+enabled categories only, for the claim form's dropdown).

Per `013`'s own framing: **Step 1 is read/written through one shared endpoint every other step also reads from (`GET`/`PATCH /api/categories/:id`); Steps 2-4 each persist through their own independent `PUT` endpoint** (`.../fields`, `.../policies`, `.../project-policies`) — there's no cross-step orchestration or transaction spanning multiple steps' saves.

**`CategoryPolicy` is one flat, `policyType`-discriminated table (`"claim" | "exception" | "project"`), not three separate tables or a nested hierarchy** — confirmed at explicit request after an early draft incorrectly nested Exception Policies under Claim Policies; the three types are fully independent lists. Claim Policies are always required (≥1); Exception Policies are always optional (0 is fine); Project Policies are required (≥1) only when `Category.enableProjectPolicies` is `true`. `category-policy-validation.ts`'s `validatePolicy(policy, options: PolicyValidationOptions)` is the one shared validator behind all three — it varies by `allowedEligibilityTypes` (Claim: department/grade/project; Exception & Project: employee/project respectively, one type each), `maxRuleLevel` (Claim: unlimited; Exception & Project: 1), and `checkRuleDuplicates`, rather than duplicating ~150 lines of eligibility/rule/approval-flow validation three times. `category-policy-persistence.ts` holds the matching shared `persistPolicy`/`buildFieldLookups` write-side helpers, split into their own util (not left inside a controller) specifically so both `category-policies.controller.ts` and `category-project-policies.controller.ts` can import them without one controller importing another.

**Two distinct replace strategies are used deliberately, not inconsistently**: `CategoryField` rows are replaced **by id** (update existing rows in place, create new ones, delete ones no longer submitted) in `saveCategoryFields`, because `CategoryPolicyRule.fieldId`/`comparisonFieldId`/`amountFieldId` and other fields' `conditionalVisibility.dependsOnFieldId` reference a field by id, and `013` requires renames to auto-propagate rather than break those references. `CategoryPolicy` rows (Step 3 and 4) use full **destroy-then-recreate** instead, since nothing outside a policy ever references its id — version snapshots store policy data by value, not by id.

**Field renames auto-update every formula/red-flag reference in the same save request** — `saveCategoryFields` builds a `renameMap` (old name → new name, matched by id between the incoming payload and the existing rows) and runs a string-substitution pass (`applyFieldRenames`) replacing `{{OldName}}` with `{{NewName}}` across every field's `config.formula` and `redFlagValue` before validating. This works only within a single save request, which is the only case `013`'s acceptance criteria actually require.

**Approval flow AND/OR semantics**: `CategoryApprovalStageApprover.logicGroup` groups approvers sharing the same integer within one stage as OR'd (any one of them suffices); different `logicGroup` values within the same stage are AND'd (every group needs its own approval). `CategoryApprovalLevel.isDefaultFlow` is the flag distinguishing the always-present "Default Flow" from the up-to-5 optional numbered Levels (`level: 1-5`, `null` when `isDefaultFlow` is true) — `validatePolicy` enforces exactly one Default Flow per policy.

**Step 4's `PUT .../project-policies` is the confirmed terminal call — there is no separate submit/activate endpoint.** Every successful call flips `Category.status` from `"draft"` to `"active"` (whether `enableProjectPolicies` is `false` with no policies, or `true` with a valid non-empty list), and — per `015`'s Edit flow — the exact same endpoint, called again later against an already-`active` category, just updates Project Policies in place without re-triggering activation (guarded by capturing `wasDraft` before mutating `status`, so version `1.0` creation only fires on the genuine draft→active transition, never on a later edit).

**016's versioning engine (`category-versioning.ts`) is triggered two ways, never per individual step-save**: `createCategoryVersion(category, createdBy)` is called (a) from `saveCategoryProjectPolicies` exactly once, on the draft→active transition, unconditionally creating `1.0` with `modifiedSteps: []` (there's no prior version to diff against); and (b) from `finishCategoryEditSession` (`POST /api/categories/:id/finish-editing`), which the frontend wizard calls once when an edit session for an *already-active* category ends (Cancel, navigate away, or close) — **this endpoint is this story's single most significant architectural resolution**: `016`'s own Open Questions flags that per-step-save versioning is structurally incompatible with the "3+ of 4 steps changed = major" rule, since no individual step's save endpoint ever sees more than its own one step's diff, so a dedicated session-end signal was added rather than inferring versioning from any existing save call. `createCategoryVersion` diffs the freshly-built current snapshot against the latest `CategoryVersion.snapshot` across all 4 step-slices (`basicDetails`/`expenseForm`/`policiesAndApprovals`/`projectPolicies`, see `WIZARD_STEPS`); 0 differing steps creates nothing, 1-2 is a minor bump (`X.Y` → `X.(Y+1)`), 3-4 is a major bump (`X.Y` → `(X+1).0`).

**Version diffing uses a recursive key-sorted `stableStringify`, not plain `JSON.stringify`** — this was a real bug caught during implementation-time verification: Postgres's `jsonb` column type does not preserve object key insertion order, so a snapshot freshly built from live relational data and that same snapshot after a round-trip through the `CategoryVersion.snapshot` JSONB column can hold identical data with differently-ordered object keys, which a naive `JSON.stringify` comparison flags as a spurious diff (observed firsthand: an edit touching only Basic Details was incorrectly computed as a major version until this was fixed). `stableStringify` sorts object keys recursively before serializing but leaves array order untouched, since every array in a `CategorySnapshot` is already meaningfully ordered (by `position`/`level`/`stageNumber` — see `category-snapshot.ts`).

`category-snapshot.ts`'s `buildCategorySnapshot(category)` is the one shared "assemble everything" builder behind `GET /api/categories/:id`, `GET .../versions/latest`, `GET .../versions/:version` (for a still-draft category, which has no real `CategoryVersion` rows), and `createCategoryVersion`'s own snapshot capture — all four read paths return the identical shape by construction, batching queries per policy type via `Promise.all` rather than issuing N+1 queries per policy/level/stage.

`013-016` introduce no new library or state-management pattern beyond what `008`'s Employee Invitation wizard already established (replace-by-id vs. destroy-then-recreate, JSONB for the first time in this codebase for `CategoryField.config`/`conditionalVisibility` and `CategoryVersion.snapshot`/`modifiedSteps`) — `OwnerRequest`/`requireOwner` was one too, until the router's gate was later removed (see above).

### Trip Management (018-019) — no `requireOwner` gate, and a real open question about `Trip.status`

`Trip`/`Country`/`City` (`trip.routes.ts`/`country.routes.ts`/`city.routes.ts`, mounted at `/api/trips`/`/api/countries`/`/api/cities`) are this codebase's first employee-facing resource that's gated by `requireAuth` alone — **no `requireOwner`**, since a trip is every employee's own personal travel record, not a Company-Administrator config screen (`018`'s own explicit posture, a deliberate departure from every other multi-row resource in this codebase). `trip.controller.ts`'s `listTrips`/`createTrip`/`deleteTrip` all scope by `employeeId: req.userId`, not `organizationId` — `organizationId` is still stored on `Trip` (for a future org-wide report), but no query in this epic ever filters by it.

`Country`/`City` are global, seeded-once, no-management-UI catalogs — the same shape `Airline` already established, right down to seeding via `bulkInsert` directly inside the creating migration (`20260709120000-create-trip-management-tables.js`) rather than a separate seeder file. **The city seed is a curated "major cities per country" list (~48 countries, ~200 cities), not a full world-cities import** — a genuinely complete dataset is tens of thousands of rows and would realistically come from a public geographic dataset (e.g. GeoNames) rather than being hand-authored; this was an explicit, flagged scope decision (see `018`'s Open Questions), not an oversight. `city.controller.ts`'s `listCities` matches search text against city name **or** country name (so searching "India" surfaces every Indian city) by first resolving matching `Country` rows, then combining `{ countryId: matchingCountryIds } OR { name: iLike }` via `Op.and`/`Op.or` composed as an array of `WhereOptions` — the same dynamic-where-conditions-array pattern `employee.controller.ts`'s `listEmployees` already uses, not a one-off.

**`Trip.status` supports four values (`"draft" | "new" | "pending_for_approval" | "approved_for_reimbursement"`) even though this epic's own code only ever writes `"new"`.** The other three exist in the type/column because `019`'s own reference screenshot showed real sample data in those states — but nothing in `018`/`019` builds what would actually produce `"pending_for_approval"`/`"approved_for_reimbursement"` (a future Claims/Approval epic's concern), and **`"draft"` specifically is a known, unresolved gap**: `018`'s Create Trip has no Save-as-Draft action, so nothing in this codebase can currently create a Draft trip, even though the delete endpoint (`deleteTrip`) already enforces "only draft trips can be deleted" defensively, and the column/type support it. Don't build a Save-as-Draft path without re-confirming this is actually wanted first — see `018`'s Open Questions.

`Trip.totalAmount`/`approvedAmount` are `DECIMAL(12,2)` columns — Sequelize returns these as strings, not numbers (avoiding float-precision issues with money), so `trip.model.ts` types them as `string`/`string | null`, not `number`. `approvedAmount` is always `null` until a future story populates it. **`totalAmount` (and `status`) are no longer untouched, though** — `utils/trip-total.ts`'s `recomputeTripFromLinkedClaims(tripId)` (added alongside Claim Management, not part of `018`/`019` themselves) sums every `Claim.totalAmount` linked to a trip via `tripId` (draft and submitted claims both count, matching how `Claim.totalAmount` itself makes no such distinction) and writes the result back to `Trip.totalAmount`; the same call also flips `Trip.status` from `"new"` to `"pending_for_approval"` the first time that sum becomes positive — a one-way transition, never reverted back to `"new"` if a claim is later deleted and the total drops back to 0. `claim.controller.ts` calls it after `saveExpenses` (whenever a trip-linked claim's own total just changed), `deleteClaim` (a trip-linked draft claim being removed), and `splitClaim` (which can leave the original and the new claim linked to two different trips, so both get recomputed independently). Since `trip-select.tsx`'s picker only ever offers `status: "new"` trips, a trip that's already picked up its first expense drops out of that picker — a deliberate consequence of the status transition, not a separate rule. `getTripDetail` also now returns a flattened `expenses` array — every `Expense` across every `Claim` with this `tripId`, one row per expense (category name, claim name, date, amount) rather than grouped by claim — read live off `Claim.tripId` rather than any new join table, so claims linked before this existed show up with no backfill needed. The frontend's own Trip Details page doesn't render the claim name column (see `frontend/CLAUDE.md`'s Trip Management section), but the API still returns it in case a future screen wants it.

`listTrips`'s `tripStartDate`/`createdDate` filters are **exact-calendar-day matches**, computed via a `dayRange(value)` helper that builds a `[startOfDay, startOfNextDay)` range and filters with `Op.gte`/`Op.lt` — not a single exact-timestamp match (which would almost never match anything, since `startAt`/`createdAt` carry a time-of-day) and not an open-ended range (the reference screenshot showed a single "Select Date" per filter, read as one calendar day — flagged as an assumption worth confirming in `019`'s Open Questions, not a definitely-final design).

### Claim Management (022-024) — a Category's own field configuration IS the expense form, a real AI/ML integration, and two amendments to already-shipped Category Management

`Claim`/`Expense` (`claim.controller.ts`, mounted at `/api/claims`) and the AI-specific pieces (`claim-ai.controller.ts`, same router) are gated by `requireAuth` alone, no `requireOwner` — same posture as Trip Management, a claim is every employee's own record. `GET /api/categories/claimable` is a narrower sibling of `013`'s own listing endpoint — active + enabled categories only, with each category's `CategoryField[]` inlined (since the claim form needs the actual field configuration to render, not just id/name). It predates Category Management's own gate being opened up (see that section above) and originally existed specifically to be reachable by any employee when `categoryRouter` was still owner-only; now that the whole router is `requireAuth`-only, that particular reason is moot, but the endpoint itself still earns its keep as a dedicated, pre-filtered read — it lives on its own tiny `categoryClaimableRouter`, mounted at the same `/categories` prefix but registered *before* `categoryRouter` in `routes/index.ts` (Express tries mounted routers in registration order; the first one whose path matches wins) — the same "narrow router mounted first" trick, not a new pattern.

**A Claim's expense form is not fixed — it's whatever the selected Category's own Step 2 field configuration says it should be.** `expense-fields.ts`'s `validateExpenseFieldValues(categoryFields, fieldValues, isDraftSave)` re-validates submitted values against the same `CategoryField[]` rows `category-fields.controller.ts`'s `validateFieldConfig` already validates at category-*configuration* time — one shared source of truth (the `CategoryField` row), two different moments it's checked. It also resolves `conditionalVisibility` (a field not currently visible is skipped entirely, not required), and recomputes any `amount`/`number` field with a `config.formula` set via `formula-evaluator.ts` — a small hand-written recursive-descent arithmetic parser (numbers, `+ - * / ( )`, `{{FieldName}}` tokens already substituted), **not `eval()`/`new Function()`**, per this codebase's security rules; the formula grammar was already constrained at category-save time, so this only has to parse that narrow subset. The function returns the three denormalized `Expense` columns (`amount`, `expenseDate`, `invoiceNumber` — read off whichever fields carry `useAsClaimAmount`/`useAsExpenseDate`/`useAsInvoiceNumber`) alongside every field's normalized value, and is reused identically by the manual flow's `saveExpenses` and the AI flow's own per-source extraction save.

**Two real amendments to already-shipped Category Management (013), both additive, no backfill needed:**
- `CategoryField.config.useAsInvoiceNumber` (boolean, optional) — mirrors `useAsClaimAmount`/`useAsExpenseDate`'s pattern but, unlike those two, **not** required to be set exactly once per category; `category-fields.controller.ts`'s non-draft save now rejects a category with *more than one* field so marked, but zero is fine (a mileage/per-diem category simply has no invoice-number concept, and duplicate detection just never checks it). Lives in the existing `config` JSONB column, same as `useAsClaimAmount`/`useAsExpenseDate` — no migration needed for the flag itself.
- **`city_list` fields now draw from Trip Management's real `Country`/`City` catalog instead of the old hardcoded 10-name `CATEGORY_CITY_LIST`**, which was deleted outright (backend copy only — the frontend's own `CATEGORY_CITY_LIST` in `field-config-panel.tsx` still needs the equivalent frontend-side change, not yet done as of this pass). `category-fields.controller.ts`'s `validateFieldConfig` now takes a `cityCount` parameter (real `City.count()`, only queried when a `city_list` field is actually present in the incoming save) instead of comparing against the old array's `.length`. Any place that *renders* a `city_list` field for value entry (the claim form) is expected to reuse Trip Management's own `CitySelect`/`GET /api/cities` — real `City.id`(s) as the value, not city-name strings; pre-existing `city_list` values saved as bare name strings (if any exist) don't map onto a real `City.id` automatically, an accepted narrow gap per `022`'s own Open Questions.

**Duplicate Bill Detection (`duplicate-expense-check.ts`'s `findDuplicateExpense`) matches on Expense Date + Invoice/Bill Number + Amount, organization-wide, across every employee — highlighted, never blocking**, confirmed at explicit request over an earlier Vendor+Invoice+Amount proposal. It returns `null` immediately if `invoiceNumber` is `null` (no field on that category carries `useAsInvoiceNumber`) rather than falling back to a weaker Date+Amount-only match — a category with no invoice-number concept structurally has nothing to check. `expenses`' own composite index `(organizationId, invoiceNumber, expenseDate, amount)` backs this without a full-table scan; `organizationId` is denormalized directly onto `Expense` (copied from the owning `Claim` at creation) specifically so this check is a single indexed lookup, not a join through `Claim` on every expense saved. The check runs at both `saveExpenses` (manual and AI flows alike, since AI-reviewed expenses are saved through the same endpoint) and its own standalone `GET /:id/expenses/:expenseId/duplicate-check` (for the AI Review screen to re-check live as the employee edits, before the final save re-checks again).

**`Expense.categoryId` is nullable** — the one real schema surprise versus `022`'s own doc, which never states this explicitly. A manually-created expense always has it set by the time `saveExpenses` persists it (parsing rejects a missing `categoryId` outright), but an AI-created expense can arrive with no confidently-suggested category (`023`'s own "organization has zero active categories" edge case, or the model simply not matching anything) and sit blank in the Review screen until the employee picks one — `saveExpenses`'s own required-category check still applies once the employee actually saves, so a claim can never be persisted past Draft with an unset category.

**The AI/ML service is a genuinely separate backend microservice (`../ai-service`), not a module inside this codebase, and it's Python, not TypeScript** — this went through two changes, not one: an earlier pass built it as an in-process `services/ai-extraction.service.ts` that called Anthropic directly; that didn't match the actual instruction ("create one separate service for AI ML"), so the real Anthropic call, its tool-forced structured output, and its own `ANTHROPIC_API_KEY`/`AI_MODEL` config were moved out to a standalone `ai-service` Node.js process. That was then rewritten from Node/TypeScript to **Python + FastAPI**, per a later explicit instruction that Python is the better fit for AI/ML work — the wire contract (`GET /api/health`, `POST /api/extract`, same request/response JSON shapes) didn't change at all, so this backend needed zero code changes for that rewrite; only `ai-service`'s own internals changed language. See that project's own `CLAUDE.md` for its Python-side architecture. `services/ai-extraction.service.ts` **here** is just a thin HTTP client — same exported `extractInvoiceData`/`CategoryForExtraction`/`AiExtractionResult` shapes throughout all of this (so `claim-ai.controller.ts` needed zero changes at its call site across either change), internally a `fetch` to `${AI_SERVICE_URL}/api/extract` with an optional `X-Internal-Api-Key` shared-secret header, not an SDK call. **If `ai-service` is unreachable or returns "not configured," the result is treated identically** — `claim-ai.controller.ts` still creates a blank `Expense` and a `failed`-status `AiExtractionLog` row (with a message distinguishing "couldn't reach the service" from "the service says it isn't configured"), so the rest of the pipeline (upload → per-source processing → Review, ready for manual fill-in) works end-to-end regardless, verified against a real running (but key-less) `ai-service` process both before and after the Python rewrite. The model is asked for field values keyed by **field name** (it has no idea what this org's numeric `CategoryField` ids are) and for the matched category by **name**; `ai-service` resolves both back to real ids before responding, so this backend never has to redo that matching. **A single invoice source is sent to the model as a native PDF document or image, never rasterized to an image ourselves** — Claude's vision API accepts PDF input directly, so `pdf-pages.ts` (built on `pdf-lib`, pure JS, no native rasterization dependency, still lives in *this* codebase since it's needed regardless of where the LLM call itself happens) only ever needs to (a) read a PDF's page count at upload time and (b) extract one or several pages into a new standalone PDF for Merge, sent as the request body to `ai-service` — a deliberate simplification versus `023`'s literal "rendered to an image" phrasing, functionally equivalent and avoiding a native-dependency rasterization library. **`ai-service` is a different runtime now (`uvicorn`, not `ts-node-dev`)** — the two dev servers used to run the identical `ts-node-dev --respawn --transpile-only src/server.ts` command, so a broad `pkill -f "ts-node-dev.*server.ts"` killed both (this bit once during implementation); that particular footgun is gone since the Python rewrite, but don't assume the two processes are otherwise interchangeable to manage.

**Every AI call is logged to `AiExtractionLog` regardless of outcome**, per explicit instruction — `runSourceExtraction` in `claim-ai.controller.ts` creates the log row (`status: "pending"`) before calling the service, then always updates it to `"completed"`/`"failed"` with whatever was captured, even on a thrown exception. **Processing runs as an unawaited background task, polled via `GET /:id/process-status`** — this codebase has no queue/worker infrastructure anywhere (the same accepted single-process limitation `employee-bulk-invite.controller.ts`'s own in-memory staging `Map` already documents), so `processInvoiceFiles` kicks off `runExtractionForClaim` without awaiting it and returns `202` immediately; the frontend's own 4-stage progress screen is expected to poll the status endpoint until every source resolves.

**Merge folds N per-page expenses into one re-extracted expense; Unmerge re-runs extraction per page from scratch rather than restoring cached values** — per `023`'s own explicit behavior. `Expense.mergedFromExpenseIds` (JSONB) is repurposed to hold the **original page numbers**, not literal expense ids, despite its name: the original per-page `Expense` rows are deleted on Merge (their ids are gone), and since Unmerge always re-extracts rather than restores, page numbers are the only thing actually needed to reconstruct the sources. Flagged here since the column name and its actual contents don't literally match — a deliberate simplification given Unmerge's own defined "re-extract, don't restore" behavior, not an oversight.

**Uploaded invoice files live on local disk**, not cloud storage — `invoice-file-storage.ts` writes to `backend/uploads/claims/<claimId>/` (gitignored), the first persisted-file feature in this codebase (`employee-bulk-invite.controller.ts`'s own uploads are processed in-memory and never written to disk). Files are served back only through an authenticated `GET /:id/invoice-files/:fileId/content` handler, never a public `express.static` path — bills carry real financial/PII data. A genuine production version would need real object storage (S3 or equivalent); this is a known, accepted gap at this app's current scale, the same category of trade-off `010`'s own in-memory upload staging already documents.

**Every existing organization is seeded with 4 sample categories** (`20260710120000-seed-default-categories.js`: Meals, Travel/Flight, Domestic Hotel, Fuel/Local Conveyance) — added after testing the AI-Powered flow surfaced an organization with zero categories, meaning nothing for the AI to match against and nothing in the Category dropdown either. Each one deliberately exercises a different mechanic covered above: Meals carries an AI-mode red flag field (the "non-veg or alcohol" example from `013`'s own doc); Travel/Flight uses `list`/`city_list`; Domestic Hotel's Amount is formula-computed from Number of Nights × Rate Per Night; Fuel/Local Conveyance deliberately has no field marked `useAsInvoiceNumber`, demonstrating duplicate detection correctly skipping a category with no invoice-number concept. **This migration only backfills organizations that already existed at the time it ran — it does not hook into `registration.controller.ts`'s `createRegistration`, so a brand-new organization created after this shipped starts with zero categories again**, the same category of gap `006`'s own default-Role seeding explicitly closed (roles are seeded both by migration *and* at registration) but this one deliberately doesn't, since auto-provisioning business configuration for every future org wasn't asked for — flagging this now rather than letting it look like an oversight later.

### Split Claim (025) — cross-employee expense sharing, not to be confused with 022's own "Split a Claim"

`025` is a genuinely different feature from `022`'s "Split a Claim" (`splitClaim` in `claim.controller.ts`) despite the near-identical name: that one moves an employee's own expenses onto a new claim they still own; `025` shares one expense's *cost* with colleagues in the same organization, who each get their own inbox entry to Accept (raising a real new claim in their own name for their own share) or Reject. To avoid the collision, `022`'s feature was renamed in code comments and its frontend button/dialog labels to **"Move to New Claim"** — no route/API path changed, since neither was ever user-facing text; `022`'s own route (`POST /:id/split`) still exists unchanged.

**One `ExpenseSplitRequest` = one original `Expense`, not a bundle of several** — a deliberate scope reduction versus that story doc's own data model (which described a request as potentially spanning multiple expenses), made because every one of the story's own reference screenshots showed exactly one expense row per request. This collapses what the doc separately called out as per-expense accept/reject and bulk accept-all/reject-all endpoints into a single accept/reject per request (`POST /api/split-requests/:id/accept`, `POST /api/split-requests/:id/reject`) — with only one expense in play, "accept this expense" and "accept everything in this request" are the same action, so the bulk endpoints were dropped rather than built as a redundant alias.

**Two tables**: `ExpenseSplitRequest` (`expenseId`, `requestedByEmployeeId`, `splitType: "percentage"|"amount"`) and `ExpenseSplitRequestMember` (`splitRequestId`, `employeeId`, `percentage`, `amount`, `isRequester`, `status: "pending"|"accepted"|"rejected"`, `respondedAt`, `resultingExpenseId`) — the requester gets their own `isRequester: true` row too (immediately `"accepted"`, no response needed), computed as the remainder after the invited colleagues' shares are subtracted, so the full breakdown (including the requester's own retained share) is always readable off one member list rather than reconstructed from the request row plus N member rows. Both tables cascade-delete off their parent `Expense`/`ExpenseSplitRequest` — deleting the original claim (still allowed while it's `"draft"`) cleans up any split requests raised against it automatically, verified during this feature's own testing.

**`createSplitRequest`** (`POST /api/claims/:id/expenses/:expenseId/split-requests`) requires the claim to still be `"draft"` and the expense to already carry a `categoryId` and a positive amount — the same gate `022`'s own Split Expense button already uses. Colleagues are validated to belong to the caller's own organization, 1-9 of them, no duplicates, no self-inclusion; percentages must sum to ≤100 (amounts to ≤ the expense's own amount), with the requester's own share computed as whatever remains. Sends one best-effort email per invited colleague (`utils/split-request-mailer.ts`, same `nodemailer` posture as `employee-invite-mailer.ts` — a send failure doesn't block the request itself) linking to `${env.corsOrigin}/claims/split-requests` (the frontend's own "Split Request" tab).

**`acceptSplitRequestExpense`/`rejectSplitRequestExpense`** (named `acceptSplitRequest`/`rejectSplitRequest` in `split-request.controller.ts`) only ever act on the caller's own non-requester member row, 404ing otherwise (so a requester can't "accept" their own request, and an outsider gets the same 404 an unauthenticated caller would). Accept requires the same Claim Type fork (`standalone`/`trip_linked`) `022`'s own claim creation uses, and creates a brand-new `Claim` (`creationMethod: "split"`, `hasBeenSaved: true`) plus one new `Expense` copying the original's category/date/invoice-number, with `fieldValues` overriding whichever field carries `useAsClaimAmount` to this member's own share — not the full original amount. 409s if the member has already responded either way.

**`listSplitRequests`** (`GET /api/split-requests?scope=received|sent`) — `received` (the default, and the only scope the frontend currently uses) returns the caller's own inbox: one row per request they were invited into, with their own status and their own share amount, not the full expense amount. `sent` returns the requester's own view (aggregate colleague count/total, not a per-member breakdown — `getSplitRequestDetail` is where the full breakdown lives). Neither the frontend's "sent" view nor its 007/024-style column filters beyond `search`/`requestedOn` were built, since 025's reference screenshots never showed one — the endpoint supports it if a future story asks.
