# CLAUDE.md

This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

## Overview

Finance Management is six independent apps in one repo with no shared workspace tooling — `frontend/`, `claim-service/`, `communications-service/`, `auth-service/`, and `gateway-service/` are Node.js (each with its own `package.json`/lockfile, installed and run separately with npm), `ai-service/` is **Python** (its own `pyproject.toml`/`uv.lock`, managed with `uv`, not npm) — each with its own `CLAUDE.md` (`claim-service/CLAUDE.md`, `frontend/CLAUDE.md`, `ai-service/CLAUDE.md`, `communications-service/CLAUDE.md`, `auth-service/CLAUDE.md`, `gateway-service/CLAUDE.md`) with service-specific commands and architecture. There is no root `package.json` (and no root Python project file either — `ai-service/` is fully self-contained). The codebase started as an early-stage skeleton (one model, one health-check route, no auth, no tests) and has grown a real feature set since — don't assume conventions beyond what's actually present in the relevant service's own `CLAUDE.md`; check for the nearest real example before inventing a pattern. `docs/PLANS/microservices-plan.md` and `docs/PLANS/microservices-frontend-integration-plan.md` track the larger, now-mostly-complete split of the original `backend/` monolith into independently-deployable services — Phases 1-4 (`communications-service`, `POST /api/auth/refresh`, `auth-service`, `claim-service` — the original `backend/` directory, renamed in place once everything else had moved out of it) are all done, and the API gateway described in `microservices-frontend-integration-plan.md` is now built (`gateway-service/`, section 1.0's "Implementation option 1" — a small Express app on `http-proxy-middleware`). What's left: Phase 5 (`reports-service`) — once it exists, add its routes to `gateway-service/src/routes/proxy-routes.ts` and it's reachable the same way everything else already is.

`gateway-service/` (the single browser-facing origin — see its own `CLAUDE.md` for the full routing table) runs on `http://localhost:4400`, `claim-service` (the renamed original `backend/`) runs on `http://localhost:4000`, frontend on `http://localhost:3000`, the AI/ML extraction microservice (`ai-service/`, Claim Management's Automated Extraction flow) on `http://localhost:4100`, `communications-service/` (email + WhatsApp delivery) on `http://localhost:4200`, and `auth-service/` (identity/org backbone — login, registration, Employee/Grade/Department/Role/Associated-Organizations/Project/Airline management) on `http://localhost:4300`. The frontend's `NEXT_PUBLIC_API_BASE_URL` points at `gateway-service` (`http://localhost:4400/api`), not at any individual service directly — `gateway-service` reads the request path and forwards it, unchanged, to whichever service owns that prefix; `auth-service`/`claim-service`/`communications-service`/`ai-service` are never called directly by the browser and don't need their own CORS config, only `gateway-service` does. `ai-service` is called only by `claim-service` over plain HTTP (`AI_SERVICE_URL`) — it's a standalone Python/FastAPI process with its own `ANTHROPIC_API_KEY`, not written in the same language as the rest of the stack: it started as an in-process Node.js module, was pulled into its own Node.js service, then rewritten to Python — Python being the better fit for AI/ML work specifically. **It now owns its own MySQL database** (`mysql-ai` in `docker-compose.yml`, port `3307`) holding `AiExtractionLog` — moved there from the monolith's Postgres in Phase 4, since that audit-log responsibility always conceptually belonged to whichever service performs the extraction. `communications-service` is called by both `claim-service` and `auth-service` over plain HTTP — it owns its own MongoDB database (`NotificationLog`, an audit trail for every email/WhatsApp send). WhatsApp has no real provider wired up yet (a deliberate, explicit decision) — `communications-service/src/services/whatsapp.service.ts` is still a console.log stub, just logged to `NotificationLog` now instead of only the console. `auth-service` owns its own Postgres database (`postgres-auth`, port `5433`) holding `Employee`/`Organization`/`Grade`/`Department`/`Role`/`AssociatedOrganization`/`Project`/`Airline`/etc. `claim-service` owns its own Postgres database (`postgres-claim`, port `5434`) holding `Category`/`Claim`/`Expense`/`Trip`/etc. Both were seeded via real `pg_dump`/restore from the original monolith's single database, not greenfield services — every table's row count was verified to match exactly before cutover. Neither service has the other's tables/models — the handful of places `claim-service` still needs employee names/emails or the `Airline` catalog call `auth-service`'s `/api/internal/*` routes instead (`claim-service/src/services/auth.service.ts`). The single biggest cross-service simplification: `organizationId` is embedded directly in the access-token JWT at issue time (safe since it's immutable per employee), so every service verifies it statelessly with zero DB/HTTP calls — this replaced a `getActiveOrganizationId` DB lookup that used to run on nearly every request across both services. **Cross-service foreign keys don't exist** — several `claim-service` migrations used to have real `references: { model: "organizations"/"employees" }` FK constraints; those were stripped to plain unenforced integer columns when `claim-service` got its own dedicated database, since a FK can't span two separate databases (same reasoning `AiExtractionLog`'s own cross-service columns already followed).

## Database

```
docker compose up -d          # start every local DB (from repo root): Postgres x3, MySQL, MongoDB
```
Each service's own `.env` (copied from its `.env.example`) must match the corresponding
`docker-compose.yml` service: `claim-service/.env` → `postgres-claim` (port `5434`), `auth-service/.env`
→ `postgres-auth` (port `5433`), `ai-service/.env` → `mysql-ai` (port `3307`),
`communications-service/.env` → `mongo` (port `27017`, no auth enabled locally).
