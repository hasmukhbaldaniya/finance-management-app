# CLAUDE.md

This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

## Overview

Finance Management is seven independent apps in one repo with no shared workspace tooling ‚Äî `frontend/`, `claim-service/`, `communications-service/`, `auth-service/`, `gateway-service/`, and `reports-service/` are Node.js (each with its own `package.json`/lockfile, installed and run separately with npm), `ai-service/` is **Python** (its own `pyproject.toml`/`uv.lock`, managed with `uv`, not npm) ‚Äî each with its own `CLAUDE.md` (`claim-service/CLAUDE.md`, `frontend/CLAUDE.md`, `ai-service/CLAUDE.md`, `communications-service/CLAUDE.md`, `auth-service/CLAUDE.md`, `gateway-service/CLAUDE.md`, `reports-service/CLAUDE.md`) with service-specific commands and architecture. There is no root `package.json` (and no root Python project file either ‚Äî `ai-service/` is fully self-contained). The codebase started as an early-stage skeleton (one model, one health-check route, no auth, no tests) and has grown a real feature set since ‚Äî don't assume conventions beyond what's actually present in the relevant service's own `CLAUDE.md`; check for the nearest real example before inventing a pattern. `docs/PLANS/microservices-plan.md` and `docs/PLANS/microservices-frontend-integration-plan.md` track the larger split of the original `backend/` monolith into independently-deployable services ‚Äî all 5 phases are now done: Phases 1-4 (`communications-service`, `POST /api/auth/refresh`, `auth-service`, `claim-service` ‚Äî the original `backend/` directory, renamed in place once everything else had moved out of it), the API gateway (`gateway-service/`), and Phase 5 (`reports-service/`, `user-stories/028-reports.md` ‚Äî three read-only org-wide reports, no database of its own, gated by `isOwner` embedded in the access-token JWT alongside `organizationId`). No further phases are planned; `user-stories/029-claim-trip-approvals.md` (Approvals) is a separate, currently on-hold epic reports-service has zero dependency on.

`gateway-service/` (the single browser-facing origin ‚Äî see its own `CLAUDE.md` for the full routing table) runs on `http://localhost:4400`, `claim-service` (the renamed original `backend/`) runs on `http://localhost:4000`, frontend on `http://localhost:3000`, the AI/ML extraction microservice (`ai-service/`, Claim Management's Automated Extraction flow) on `http://localhost:4100`, `communications-service/` (email + WhatsApp delivery) on `http://localhost:4200`, `auth-service/` (identity/org backbone ‚Äî login, registration, Employee/Grade/Department/Role/Associated-Organizations/Project/Airline management) on `http://localhost:4300`, and `reports-service/` (no database of its own ‚Äî forwards the caller's cookie to `auth-service`/`claim-service`'s org-wide read endpoints and aggregates in memory) on `http://localhost:4500`. The frontend's `NEXT_PUBLIC_API_BASE_URL` points at `gateway-service` (`http://localhost:4400/api`), not at any individual service directly ‚Äî `gateway-service` reads the request path and forwards it, unchanged, to whichever service owns that prefix; `auth-service`/`claim-service`/`communications-service`/`ai-service` are never called directly by the browser and don't need their own CORS config, only `gateway-service` does. `ai-service` is called only by `claim-service` over plain HTTP (`AI_SERVICE_URL`) ‚Äî it's a standalone Python/FastAPI process with its own `ANTHROPIC_API_KEY`, not written in the same language as the rest of the stack: it started as an in-process Node.js module, was pulled into its own Node.js service, then rewritten to Python ‚Äî Python being the better fit for AI/ML work specifically. **It now owns its own MySQL database** (`mysql-ai` in `docker-compose.yml`, port `3307`) holding `AiExtractionLog` ‚Äî moved there from the monolith's Postgres in Phase 4, since that audit-log responsibility always conceptually belonged to whichever service performs the extraction. `communications-service` is called by both `claim-service` and `auth-service` over plain HTTP ‚Äî it owns its own MongoDB database (`NotificationLog`, an audit trail for every email/WhatsApp send). WhatsApp has no real provider wired up yet (a deliberate, explicit decision) ‚Äî `communications-service/src/services/whatsapp.service.ts` is still a console.log stub, just logged to `NotificationLog` now instead of only the console. `auth-service` owns its own Postgres database (`postgres-auth`, port `5433`) holding `Employee`/`Organization`/`Grade`/`Department`/`Role`/`AssociatedOrganization`/`Project`/`Airline`/etc. `claim-service` owns its own Postgres database (`postgres-claim`, port `5434`) holding `Category`/`Claim`/`Expense`/`Trip`/etc. Both were seeded via real `pg_dump`/restore from the original monolith's single database, not greenfield services ‚Äî every table's row count was verified to match exactly before cutover. Neither service has the other's tables/models ‚Äî the handful of places `claim-service` still needs employee names/emails or the `Airline` catalog call `auth-service`'s `/api/internal/*` routes instead (`claim-service/src/services/auth.service.ts`). The single biggest cross-service simplification: `organizationId` is embedded directly in the access-token JWT at issue time (safe since it's immutable per employee), so every service verifies it statelessly with zero DB/HTTP calls ‚Äî this replaced a `getActiveOrganizationId` DB lookup that used to run on nearly every request across both services. **Cross-service foreign keys don't exist** ‚Äî several `claim-service` migrations used to have real `references: { model: "organizations"/"employees" }` FK constraints; those were stripped to plain unenforced integer columns when `claim-service` got its own dedicated database, since a FK can't span two separate databases (same reasoning `AiExtractionLog`'s own cross-service columns already followed).

## Database

```
docker compose up -d          # start every local DB (from repo root): Postgres x3, MySQL, MongoDB
```
Each service's own `.env` (copied from its `.env.example`) must match the corresponding
`docker-compose.yml` service: `claim-service/.env` ‚Üí `postgres-claim` (port `5434`), `auth-service/.env`
‚Üí `postgres-auth` (port `5433`), `ai-service/.env` ‚Üí `mysql-ai` (port `3307`),
`communications-service/.env` ‚Üí `mongo` (port `27017`, no auth enabled locally).
